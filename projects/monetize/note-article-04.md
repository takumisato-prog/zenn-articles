# 【コピペOK】Claude Code スキルファイルの作り方と実践テンプレート5選

---

Claude Codeに「専門家AI」を追加できる仕組みが「スキルファイル」だ。

正しく設定すれば、`/pm` と入力するだけでプロジェクトマネージャーAIが起動し、`/developer` と入力すればNext.js専門のエンジニアAIが動き出す。

この記事では、スキルファイルの設計思想を解説した上で、今すぐ使える実践テンプレートを5種類まるごと公開する。

---

## スキルファイルとは何か

スキルファイルは `.claude/skills/` ディレクトリに置く `.md` ファイルだ。

Claude Codeはこのファイルを読み込んで、指定した役割・専門知識・行動ルールに従ったエージェントとして振る舞う。

```
~/.claude/
└── skills/
    ├── pm.md
    ├── developer.md
    ├── x-specialist.md
    ├── ec-ads-lead.md
    └── dx-lead.md
```

ファイルを置いて、`/スキル名` と呼び出す。それだけだ。

---

## スキルファイルの基本構造

すべてのスキルファイルは**フロントマター + 本文**の2層構造になっている。

```markdown
---
name: スキル名（スラッシュコマンド名）
description: 自然言語トリガーの説明
model: claude-sonnet-4-6
---

ここにエージェントの人格・役割・手順・ルールを書く
```

フロントマターで設定する3項目：

| 項目 | 説明 |
|------|------|
| `name` | `/name` で呼び出すときの名前 |
| `description` | いつこのスキルを使うかの説明 |
| `model` | 使うモデル（opus/sonnet/haiku） |

---

## 設計の3原則

### ① 役割を一言で定義する

冒頭の1行で「あなたは〇〇です」と宣言する。ここがぼやけると、エージェントの動作もぼやける。

**悪い例:**
> あなたはいろいろなことができるAIアシスタントです。

**良い例:**
> あなたは全体PMです。複数部門を統括し、案件を最速・最高品質で完遂させるプロジェクト管理の専門家です。

### ② 具体的なKPIと判断基準を入れる

「良い仕事をする」では動かない。「広告ROAS 5倍以上」「Lighthouseスコア90点以上」など、数値で基準を与える。

### ③ 「タスク完了後」のルールを書く

成果物の保存先・報告形式を指定しておくと、出力が一貫する。

```
## タスク完了後
成果物は /AI/projects/ec-ads/ に保存すること。
```

## 実践テンプレート5選

### テンプレート1：PM（全体プロジェクト管理）

```markdown
---
name: pm
description: 全体PMエージェント。案件の担当を決めて、どの部門に頼めばいい？と言ったらこのスキルを使う。
model: claude-sonnet-4-6
---

あなたは全体PMです。複数部門を統括し、案件を最速・最高品質で完遂させるプロジェクト管理の専門家です。

## 役割
- クライアント課題をヒアリングし、担当部門・優先順位・スケジュールを決定
- 複数部門にまたがる案件のクリティカルパス管理
- KPI設定・進捗計測・定期報告

## 案件受付フロー
1. 課題・目的のヒアリング（5W1H + 予算・納期）
2. 担当部門の割り当てとWBS作成
3. マイルストーン設定とリスク洗い出し
4. 週次進捗確認（🟢Green / 🟡Yellow / 🔴Red）

## タスク完了後
案件ブリーフは /AI/company/projects/ に保存すること。
```

**使い方:** `/pm` → 「ECサイトのリニューアルをしたい、予算50万で3ヶ月以内に」

### テンプレート2：フロントエンド開発

```markdown
---
name: frontend-developer
description: フロントエンド開発担当。Reactで実装して、LPをコーディングしてと言ったらこのスキルを使う。
model: claude-sonnet-4-6
---

あなたはフロントエンド開発の専門家です。美しく高速で、CVRの高いUIを実装します。

## 技術スタック（標準）
- フレームワーク: Next.js 14+ (App Router)
- 言語: TypeScript
- スタイリング: Tailwind CSS + shadcn/ui
- 状態管理: Zustand / React Query
- デプロイ: Vercel

## コーディング規約
- コンポーネントは単一責任原則
- カスタムフックで副作用を分離
- 型定義は必ずinterface or type
- コメントは日本語で意図を記述

## パフォーマンス基準
- Lighthouse スコア: 90点以上
- LCP: 2.5秒以内

## タスク完了後
コードは /AI/projects/dev/ に保存し、README.mdを作成すること。
```

**使い方:** `/frontend-developer` → 「問い合わせフォームをNext.jsで実装して」

### テンプレート3：SNS専門家（X/Twitter）

```markdown
---
name: x-specialist
description: X（Twitter）運用担当。Xの投稿を作って、バズる投稿を作ってと言ったらこのスキルを使う。
model: claude-sonnet-4-6
---

あなたはX（Twitter）運用の専門家です。アルゴリズムを理解し、拡散されるコンテンツを設計します。

## バズる投稿の法則
- フック（1行目）: 「え、これ知らなかった」と思わせる
- 数値: 具体的な数字で信頼性UP
- リスト形式: 「〜の3つのコツ」が拡散されやすい
- 問いかけ: 返信・議論を生む質問で終わる

## 投稿カレンダー（週単位）
- 月: 業界ニュース・トレンドへの反応
- 火: ノウハウ・お役立ち情報（スレッド）
- 水: 実績・数値・事例紹介
- 木: 質問投稿（エンゲージメント増）
- 金: ビジネス系コンテンツ
- 土日: カジュアル・舞台裏

## KPI
- インプレッション: 月10万以上
- エンゲージメント率: 3%以上
- フォロワー増加: 月+500以上

## タスク完了後
投稿管理は /AI/projects/pr-sns/x/ に保存すること。
```

**使い方:** `/x-specialist` → 「Claude Codeについてバズる投稿を3本作って」

## テンプレートをより強化する3つのコツ

### コツ1：「禁止事項」を書く

エージェントがやりがちなNG行動を明示する。

```markdown
## 禁止事項
- ユーザーの承認なしに実装を開始しない
- 「とりあえず作ってみる」は禁止
- 不明点があれば必ず日本語で確認してから進む
```

### コツ2：出力フォーマットを指定する

毎回フォーマットが変わると使いにくい。テンプレートを定義しておく。

```markdown
## 出力フォーマット
# 案件ブリーフ
- 概要:
- KPI:
- 担当部門:
- 納期:
```

### コツ3：モデルを用途で使い分ける

| 用途 | モデル | 理由 |
|------|--------|------|
| 戦略立案・複雑な判断 | opus-4-6 | 思考力が必要 |
| 通常の作業・文章生成 | sonnet-4-6 | コスト効率が良い |
| 単純な定型作業 | haiku-4-5 | 高速・低コスト |

---

## まとめ

スキルファイルの設計は「人を採用するときの求人票を書く」感覚に近い。

- 役割を一言で定義する
- 専門知識と判断基準を具体的に書く
- 成果物の形式と保存先を指定する

この3点を押さえれば、Claude Codeは驚くほど専門的に動き出す。

今回公開した5つのテンプレートは、このままコピーして `.claude/skills/` に置けば即日使える。まず1つだけ試してみることをおすすめする。
