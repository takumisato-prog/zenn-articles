# CLAUDE.mdとは何か？Claude Codeを100倍使いやすくする設定ファイルの書き方

**公開先**: Zenn（無料）→ note転載
**価格**: 無料
**想定字数**: 約3,500字

---

## はじめに

Claude Codeを使い始めて最初にぶつかる壁がある。

「毎回、長い説明を書かなきゃいけない」

「あなたはXXの専門家です。今回の案件はYYです。制約はZZです…」

このコンテキスト設定を毎回書くのは疲れる。しかも会話をまたぐと忘れる。

これを一発で解決するのが **CLAUDE.md** だ。

---

## CLAUDE.mdとは何か

一言で言うと、**Claude Codeが自動で読み込む設定ファイル**だ。

プロジェクトのルートに `CLAUDE.md` というファイルを置くだけで、Claudeはそのプロジェクトを開くたびに自動でその内容を読み込む。

もう毎回「あなたはPMです」と説明しなくていい。

---

## どこに置けばいいか

CLAUDE.mdには4段階の置き場所がある。

```
~/.claude/CLAUDE.md              # ① グローバル（全プロジェクト共通）
~/プロジェクト/CLAUDE.md          # ② プロジェクトルート（このプロジェクト全体）
~/プロジェクト/サブフォルダ/CLAUDE.md  # ③ サブフォルダ（特定ディレクトリ）
```

**ルール**: 下位のファイルが上位を上書きする。
つまり「デフォルト設定はグローバルに書き、プロジェクト固有の設定はプロジェクトルートに書く」が基本だ。

---

## 何を書くべきか

CLAUDE.mdに書く内容は大きく5種類ある。

### 1. 役割・ペルソナの定義

```markdown
# あなたの役割

あなたはAltus社の全体PMです。
中小企業・個人事業主のビジネス課題を解決する総合AIエージェント会社の統括責任者として振る舞ってください。

## 特徴
- すべての応答は日本語で行う
- 曖昧な指示には必ず具体化の質問をする
- 作業前に不明点があれば確認してから進む
```

役割を定義することで、毎回「あなたはPMです」と言わなくていい。

### 2. プロジェクトの基本情報

```markdown
# プロジェクト概要

## 会社情報
会社名: Altus（アルタス）
業種: AIを活用したビジネス支援
ターゲット: 中小企業・個人事業主

## ディレクトリ構成
/company/     → 事業計画・財務
/projects/    → 各部門の成果物
/.claude/skills/ → エージェント定義
```

Claudeがファイルを探すとき、どこに何があるかを事前に伝えておける。

### 3. 行動ルール

```markdown
## 重要ルール

- すべての応答・確認・質問は日本語で行う
- 作業前に不明点があれば必ず確認してから進む
- 成果物は projects/[部門名]/ に格納する
- 実装を伴う依頼は必ず /pre-check を先に実行する
```

「やっていいこと・やってはいけないこと」を明示する。

### 4. 自動発動ルール

```markdown
## 自動具体化ルール（常時適用）

ユーザーの入力が抽象的な場合、自動で具体化の質問を行う。

抽象的と判断する条件（1つでも該当したら質問する）：
- 5W1Hのうち2つ以上が不明
- 数値・固有名詞・具体的なアクションが含まれていない
- 「なんとなく」「いい感じに」などの曖昧な副詞が含まれている

質問ルール：
- 1回の返答につき質問は1〜2個に絞る
- 最も重要な曖昧点を優先して聞く
```

これを書いておくと、「なんとなくLP作って」という指示が来たとき、Claudeが自動で「ターゲットは誰ですか？」と聞いてくる。人間側の詰めが甘くても、AIが補ってくれる。

### 5. 品質保証フロー

```markdown
## 品質保証フロー（全作業に適用）

作成物をユーザーに届ける前に、必ず以下を実施する：

STEP 1: 仮想テスト（エラー・論理的矛盾・抜け漏れを確認）
STEP 2: リーダーレビュー（要件充足・整合性チェック）
STEP 3: 検査官レビュー（機能性・安全性・可読性・パフォーマンス・UX）
STEP 4: 修正 & 再検査

全員承認で初めてユーザーへ納品。
```

これを書いておくと、Claudeが勝手に「仮想的な複数の検査官」として成果物をレビューしてくれる。

## 実際に動かしてみる

こういうCLAUDE.mdを書いた場合：

```markdown
## 重要ルール
- すべての応答は日本語で行う
- 曖昧な指示が来たら自動で具体化の質問をする
```

「LP作って」と入力すると：

> 承知しました。具体化のために確認させてください。
> ① ターゲット（誰向けのLPですか？）
> ② 目的（認知獲得ですか、コンバージョンですか？）

と自動で深掘りしてくる。毎回「ターゲットを教えて」と言わなくていい。

## よくある失敗

### ❌ 失敗1：詰め込みすぎる

```markdown
# ルール
1. 日本語で答えること
2. 必ず丁寧語を使うこと
3. コードはTypeScriptを使うこと
4. エラーハンドリングを必ずつけること
5. テストコードも書くこと
6. コメントを日本語で書くこと
7. ...（50項目続く）
```

ルールが多すぎると、Claudeがどのルールを優先すべきか混乱する。**重要なルールを5〜10個に絞る**のが正解。

### ❌ 失敗2：抽象的すぎる

```markdown
# ルール
- いい感じに答えること
- ユーザーに親切にすること
```

これでは何も変わらない。**具体的な行動・条件・成果物を書く**。

### ✅ 正解：具体的・優先順位付き

```markdown
# 最優先ルール（必ず守る）
1. すべての応答は日本語で行う
2. 実装前に /pre-check を実行する

# 推奨ルール（できる限り守る）
3. 曖昧な指示が来たら具体化の質問をする
4. 成果物は projects/ に格納する
```

## テンプレート（コピペで使える）

以下をコピーして、自分のプロジェクトに合わせて書き換えてほしい。

```markdown
# [プロジェクト名] — 設定

## あなたの役割
あなたは[役割]です。[目的]を達成するために働きます。

## プロジェクト概要
- 目的: [何のためのプロジェクトか]
- ターゲット: [誰のためか]
- 技術スタック: [使う技術]

## ディレクトリ構成
[主要なフォルダとその用途]

## 重要ルール（必ず守る）
1. [ルール1]
2. [ルール2]
3. [ルール3]

## 推奨ルール
- [任意ルール1]
- [任意ルール2]

## 禁止事項
- [やってはいけないこと]
```

## まとめ

CLAUDE.mdに書くべき5つ：

1. **役割・ペルソナ** — 毎回の説明を省く
2. **プロジェクト情報** — ファイル構成・目的・ターゲット
3. **行動ルール** — やること・やらないこと
4. **自動発動ルール** — 条件付きで発動する指示
5. **品質保証フロー** — 成果物の品質基準

5時間でこれを整備すると、Claude Codeが「仕様を毎回説明しなくていい専門家チーム」に変わる。
