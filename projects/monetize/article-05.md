

# Claude Codeで自動具体化システムを作った——曖昧な指示をAIが自動で深掘りする仕組み

## はじめに

「いい感じにLP作って」

「なんかブログ記事書いて」

「売上上がる施策考えて」

——こういう指示をAIに投げて、返ってきた成果物に「なんか違うんだよな…」と思った経験はないだろうか？

自分は何十回もあった。そしてある日気づいた。**問題はAIの能力ではなく、自分の指示が曖昧すぎること**だった。

ただ、人間は本質的に「曖昧な生き物」だ。頭の中にぼんやりとしたイメージはあるが、それを5W1Hで完璧に言語化してからAIに投げる人は少ない。そもそも、それができるなら最初から自分でやれる。

だから逆転の発想で考えた。**人間がAIに質問するのではなく、AIが人間に質問する仕組みを作ればいい。**

CLAUDE.mdに「自動具体化ルール」を書き込み、曖昧な指示が来たらClaude自身が自動で深掘りしてくれるシステムを構築した。この記事では、その設計思想と実装方法を全て公開する。

---

## なぜ「AIに質問させる」のか——逆転の発想の原点

多くの人がAIを「答えを出す機械」として扱っている。質問する→答えが返る。このワンウェイの関係だ。

しかし、優秀な人間のコンサルタントやデザイナーを思い出してほしい。彼らは最初に何をするか？ **ヒアリング**だ。いきなり成果物を作り始める専門家はいない。

「ターゲットは誰ですか？」「予算感は？」「競合はどこを意識してますか？」「納期はいつですか？」

こうした質問を通じて、クライアント自身も気づいていなかった要件を引き出す。これがプロの仕事だ。

あなたはAIに「プロとしてのヒアリング」をさせているだろうか？ おそらくほとんどの人がさせていない。

自分もそうだった。Claude Codeを使い始めた当初、「指示は詳細に書くべき」という思い込みがあった。だから毎回300字くらいの長文プロンプトを書いていた。それでも成果物の精度は60〜70点だった。

転機は、CLAUDE.mdに「曖昧な指示には質問を返せ」と一行書いたときだ。たったそれだけで、成果物の精度が体感で90点以上に跳ね上がった。人間が頑張って詳細に書くより、AIに足りない情報を聞かせたほうが圧倒的に早くて正確だったのだ。

## 自動具体化ルールの設計思想——3つの原則

自動具体化システムを設計する上で、自分が定めた原則は3つある。

**原則1: 検出は厳密に、質問は最小限に**

何でもかんでも質問を返すAIは使いにくい。「今日の天気は？」に対して「どの地域ですか？何時の天気ですか？気温も必要ですか？」と返されたら鬱陶しい。曖昧さの「検出条件」を明確に定義し、かつ1回の質問は1〜2個に絞る。これが快適さと精度の両立に必要だ。

**原則2: 質問には必ず「選択肢」か「デフォルト値」をつける**

「ターゲットは誰ですか？」と聞かれても、答えに困ることがある。「ターゲットはA) 20代女性、B) 30代ビジネスマン、C) 個人事業主、のどれに近いですか？ 特になければ"中小企業の経営者"をデフォルトにします」と聞かれたほうが100倍答えやすい。AIの質問にも「UX」が必要だ。

**原則3: 具体化が完了したら、作業前に「要件サマリー」を提示する**

質問を2〜3回繰り返して情報が揃ったら、いきなり作業に入るのではなく「こういう理解で合ってますか？」と確認を挟む。人間のプロジェクトマネージャーが要件定義書を作るのと同じだ。

この3原則を守ることで、「質問が多くてウザい」と「曖昧なまま進んで手戻りが発生する」の間の最適なバランスを実現できた。

## CLAUDE.mdへの実装——コピペで使える自動具体化ルール

ここからが本題だ。自分がCLAUDE.mdに実際に書いている自動具体化ルールの全文を公開する。

```markdown
## 自動具体化ルール（常時適用）

ユーザーの入力が「抽象的」と判断された場合、作業に入る前に自動で具体化の質問を行う。

### 抽象的と判断する条件（以下のうち1つでも該当したら質問する）

1. **5W1Hのうち2つ以上が不明**
   - Who: 誰に向けたものか
   - What: 具体的に何を作るのか
   - Why: 何のために（目的・KPI）
   - When: いつまでに・いつ使うのか
   - Where: どこで使うのか（媒体・プラットフォーム）
   - How: どのような手段・フォーマットで

2. **数値・固有名詞・具体的アクションが含まれていない**
   - × 「売上を上げたい」 → ○ 「月商100万円を150万円にしたい」
   - × 「記事を書いて」 → ○ 「Zennに3,000字のClaude Code解説記事を書いて」

3. **曖昧な修飾語が含まれている**
   - 「いい感じに」「なんとなく」「適当に」「うまく」「ちゃんと」

### 質問のルール

- **1回の返答につき質問は最大2個**に絞る（それ以上は圧迫感が出る）
- 最も重要な曖昧点を優先して聞く（優先順位: Why > Who > What > How > When > Where）
- **必ず選択肢またはデフォルト値を提示する**
  - 「ターゲットは A) 20代個人 B) 中小企業経営者 C) その他 のどれに近いですか？
     特に指定がなければBで進めます」
- ユーザーが「任せる」「おまかせ」と答えた場合はデフォルト値を採用して進む
- **質問は最大2ラウンドまで**。3回目以降は持っている情報で最善の判断をして作業に入る

### 具体化完了後のフロー

質問で情報が揃ったら、作業に入る前に以下の「要件サマリー」を提示する:

"""
📋 要件サマリー
- 目的: ○○
- ターゲット: ○○
- 成果物: ○○
- 条件/制約: ○○
- 納品形式: ○○

→ この内容で進めてよいですか？
"""

ユーザーがOKを出してから作業開始。
```

これをCLAUDE.mdに貼るだけで、Claude Codeの振る舞いが劇的に変わる。曖昧な指示を投げると、Claudeが自動的に「ちょっと待ってください、確認させてください」と返してくるようになる。

## 実際のやり取り——ビフォーアフター

では、このルールがあるとないとで何が変わるのか。実際のやり取りで比較する。

### ビフォー（自動具体化ルールなし）

```
ユーザー: 「LPを作って」

Claude: 「はい、LPを作成します。以下がLP案です——
  ヘッダー: あなたのビジネスを加速する
  サブヘッド: 今すぐ始めましょう
  CTA: お問い合わせはこちら
  ...」
```

一見ちゃんと動いている。しかし「誰向け？」「何のサービス？」「コンバージョンは何？」が全て不明のまま、ふわっとしたテンプレLPが出てくる。使い物にならない。

### アフター（自動具体化ルールあり）

```
ユーザー: 「LPを作って」

Claude: 「LPを作成しますね。その前に2点確認させてください。

① このLPの目的は何ですか？
   A) 問い合わせ獲得  B) メルマガ登録  C) 商品購入  D) その他
   特に指定がなければ A) 問い合わせ獲得 で進めます。

② ターゲットは誰ですか？
   A) 個人（20〜30代）  B) 中小企業経営者  C) フリーランス  D) その他
   特に指定がなければ B) 中小企業経営者 で進めます。」

ユーザー: 「Bの商品購入で、ターゲットは副業やってる会社員」

Claude: 「ありがとうございます。もう1点だけ確認です。
販売する商品/サービスの名前と価格帯を教えていただけますか？
（例: オンライン講座・月額9,800円 など）」

ユーザー: 「Claude Code活用講座、買い切り29,800円」

Claude: 「📋 要件サマリー
- 目的: 商品購入（CV = 購入ボタンクリック）
- ターゲット: 副業中の会社員（30代前後を想定）
- 商品: Claude Code活用講座（買い切り29,800円）
- 成果物: LP（HTML）
- トーン: 専門性と親しみやすさのバランス

→ この内容で進めてよいですか？」
```

わずか2ラウンドの質問で、成果物の精度が跳ね上がる。しかも自分（ユーザー側）は「LP作って」と5文字打っただけだ。面倒な詳細指定はAIが引き出してくれる。

## 設計のコツ——自分が試行錯誤して辿り着いた5つの知見

このシステムを約2ヶ月運用して、改善を重ねてきた。その中で得た知見を5つ共有する。

**1. 質問の優先順位は「Why > Who」が鉄板**

最初は「What（何を作るか）」を最優先で聞いていたが、これだと表面的な成果物しか出ない。「なぜそれが必要か（Why）」を先に聞くことで、ユーザー自身も気づいていなかった本質的な課題が見えてくる。「LP作って」の裏にある「リード獲得を3倍にしたい」が分かれば、LP以外の提案もできるようになる。

**2. 「質問は最大2ラウンドまで」の制限が重要**

最初はこの制限を入れていなかった。結果、Claude が5回も6回も質問を繰り返し、「もういいから早く作ってくれ」とイライラした。2ラウンド制限を入れてからは、Claudeが「最も重要な質問」を自分で選別するようになった。制約が判断力を生むという発見だった。

**3. 「任せる」への対応を必ず書く**

ユーザーが「任せる」「おまかせ」と返してくることがある。ここへの対応を書いていないと、Claudeがフリーズするか、もう一度同じ質問を繰り返す。「デフォルト値で進む」というルールを明記しておくだけで、この問題は解消される。

**4. 要件サマリーは「確認」であって「承認フロー」ではない**

最初は要件サマリーをガチガチのフォーマットで出していた。しかし、ちょっとした依頼にまで重厚なサマリーが出ると面倒になる。「3行以内の軽い確認」にしたほうが運用が回る。あなたは自分のワークフローに合わせてサマリーの粒度を調整してほしい。

**5. 「抽象的でない指示」にはそのまま動いてもらう**

ここが意外と盲点だ。「Zennに3,000字でClaude Codeのスキルファイル解説記事を書いて。ターゲットはエンジニア初心者」——この指示は十分に具体的だ。これに対して質問を返されると逆にストレスになる。検出条件を明確にして、具体的な指示にはそのまま作業に入ってもらう設計が必須だ。

## さらに発展させる——部門別の具体化テンプレート

自動具体化ルールは「汎用版」だが、部門ごとにカスタマイズするとさらに精度が上がる。自分が使っている部門別の質問テンプレートの一部を公開する。

```markdown
## 部門別 具体化テンプレート

### コピーライティング部門
不明時に優先して確認する項目:
1. 誰に向けた文章か（ペルソナの年齢・職業・悩み）
2. 読んだ後にどうなってほしいか（行動変容のゴール）
3. トーン（論理的/感情的、フォーマル/カジュアル）
4. 文字数・掲載媒体

### 開発部門
不明時に優先して確認する項目:
1. 何を解決するツール/スクリプトか（課題の具体化）
2. 使用言語・フレームワークの指定
3. 入力と出力の形式（何を入れたら何が出るか）
4. エラー時の挙動（止める/ログだけ出す/スキップ）

### EC・広告部門
不明時に優先して確認する項目:
1. 商品名・カテゴリ・価格帯
2. 広告プラットフォーム（Amazon広告/Google/Meta）
3. 月間広告予算
4. 目標ROAS・目標ACoS
```

このように部門ごとの「聞くべき項目」を定義しておくと、PMエージェントが案件を振り分けた後、各部門が適切な深掘りを自動で行ってくれる。人間は最初に5文字のざっくり指示を投げるだけで、AIチームが勝手に要件を固めてくれる世界だ。

## まとめ——AIの使い方は「指示の精度」から「仕組みの設計」へ

「プロンプトエンジニアリング」という言葉がある。AIへの指示を工夫することで精度を上げる技術だ。もちろん重要だが、自分は**毎回プロンプトを工夫するのは持続可能ではない**と思っている。

やるべきは、CLAUDE.mdに仕組みを一度書いてしまうことだ。そうすれば、雑な指示を投げても、AIが自動で深掘りし、要件を固め、精度の高い成果物を出してくれる。

もう一度整理する。

1. **自動具体化ルール**をCLAUDE.mdに書く
2. 曖昧な指示が来たらClaudeが**自動で質問**を返す
3. 質問は**最大2ラウンド・各2問以内**に制限する
4. 情報が揃ったら**要件サマリー**を提示して確認
5. OKが出たら作業開始

この5ステップを仕組み化するだけで、AIとの協業の質が根本的に変わる。

「AIが思った通りの成果物を出してくれない」と嘆く前に、AIに質問させる仕組みを作ってみてほしい。問題は、AIの能力ではなく、情報の引き出し方にある。
