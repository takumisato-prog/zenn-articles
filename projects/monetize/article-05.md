# Claude Codeで自動具体化システムを作った——曖昧な指示をAIが自動で深掘りする仕組み

## 「とりあえずやっといて」がAIを壊す

Claude Codeを使い始めて最初にぶつかる壁がある。

「ユーザー登録機能を作って」

この一言を投げた瞬間、AIは何かを作り始める。でも出てきたものを見ると、「あ、そういう意味じゃなかった」が続発する。メールアドレスだけでいいと思っていたのに、電話番号フィールドまで作られている。パスワードの強度チェックは不要だったのに、複雑なバリデーションが入っている。

結果として、作り直しのループが始まる。

自分もこれを何度も経験して、あるときふと思った。**「最初から正確な仕様を渡せればいいのに」と考えるより、「AIに足りない情報を聞かせればいいんじゃないか」** と。

これが、自分がCLAUDE.mdに「自動具体化ルール」を実装するきっかけだった。

---

## 逆転の発想——AIに質問させる

普通の使い方は「人間が指示する→AIが動く」だ。

でも自分が実装したのは、このフローを変えることだった。

```
人間が曖昧な指示をする
    ↓
AIが不足情報を特定する
    ↓
AIが人間に質問する
    ↓
人間が答える
    ↓
AIが具体化された仕様で動く
```

AIを「指示を受ける存在」から「情報を引き出す存在」に変える。これが自動具体化システムの核心だ。

---

## CLAUDE.mdへの実装

実際にCLAUDE.mdに書いたルールを見てもらう。

```markdown
## 自動具体化ルール

### 原則
タスクを受け取ったとき、即座に実行しないこと。
まず「具体化フェーズ」を実行する。

### 具体化フェーズの手順

1. **受け取ったタスクの曖昧な部分を列挙する**
   - 誰が使うか（ユーザーの種類・権限）
   - いつ使うか（頻度・タイミング）
   - どこで使うか（画面・環境）
   - 何と連携するか（既存システム・API）
   - 制約条件（パフォーマンス・セキュリティ）

2. **質問を優先度順に並べる**
   - 必須（これがないと着手できない）
   - 重要（品質に大きく影響する）
   - 任意（後からでも変更できる）

3. **必須の質問のみ聞く**
   - 一度に聞く質問は最大3つまで
   - 箇条書きで簡潔に
   - 回答例を添える

### 例外
以下の場合は具体化フェーズをスキップしてよい
- 「今すぐやって」「とりあえず」という言葉がある
- 前の会話で同じタスクの仕様が出ている
- 修正・デバッグ系のタスク
```

このルールがあると、AIの動き方が変わる。

## 実際の動作例

たとえばこんな指示を投げると：

```
ユーザーに通知を送る機能を作りたい
```

以前は何も確認せずにメール送信コードが生成されていた。

でも今はこう返ってくる：

```
具体化フェーズを実行します。

以下の3点を確認させてください。

【必須】
1. 通知の手段は何ですか？
   例：メール / アプリ内通知 / Slack / その他

2. どのタイミングで通知しますか？
   例：特定のイベント発生時 / 定期的 / 手動トリガー

3. 通知する対象ユーザーの範囲は？
   例：全ユーザー / 特定の権限を持つユーザー / 個別指定

これらが決まれば、具体的な実装に入れます。
```

これに答えると、今度は仕様が明確な状態で実装が始まる。手戻りがほぼなくなった。

## 具体化の深度を制御する

最初は単純な質問リストだったが、使い込むうちに「具体化の深度」という概念が必要だと気づいた。

小さなタスクに5つも質問されると、逆にストレスになる。大きなタスクなら10個の質問でも必要かもしれない。

そこでCLAUDE.mdにタスクサイズの判定ロジックを追加した：

```markdown
## タスクサイズの判定と質問数の制御

### サイズ判定基準

**Small（質問0〜1個）**
- 修正・調整系（色を変える、テキストを変える）
- 単一ファイルの変更
- 明確な完了条件がある

**Medium（質問1〜3個）**
- 新機能の追加
- 複数ファイルにまたがる変更
- 外部連携を含む

**Large（質問3〜5個）**
- 新しいシステム・モジュールの設計
- アーキテクチャに影響する変更
- 複数人が使う機能

### 判定できない場合
Mediumとして扱う。
```

このルールを追加してから、「小さいことで無駄に質問される」ストレスがなくなった。

## 具体化の結果を記録する仕組み

もう一つ実装したのが、具体化の結果を記録する仕組みだ。

会話の中で決まった仕様は、AIが会話の中に持つだけで、次のセッションでは消える。同じ質問を2回されるのは非効率だ。

そこでこういうルールを追加した：

```markdown
## 具体化結果の記録

タスクの具体化が完了したら、以下の形式でまとめを出力する。

---
## タスク仕様メモ

**タスク名**: [タスク名]
**日付**: [今日の日付]

### 決定した仕様
- [決定事項1]
- [決定事項2]

### 前提条件
- [前提1]
- [前提2]

### 対象外（スコープ外）
- [対象外1]

---

このメモはDOCS/specs/[タスク名].mdとして保存すること。
```

これで仕様が蓄積されていく。後から「あのとき何を決めたっけ」というときに参照できる。プロジェクトのWikiとして機能し始めた。

## 設計思想——なぜこれが機能するか

このシステムが機能する理由を少し整理しておく。

**1. AIは「何が足りないか」の判断が得意**

AIは膨大なコードと仕様書を学習している。だから「この指示で何が足りないか」を判断するのが得意だ。人間より早く「この情報がないと困る」に気づける。

**2. 人間は「答える」方が「考える」より簡単**

「ユーザー登録機能を設計してください」と言われると思考が止まる。でも「メール認証は必要ですか？」と聞かれると即答できる。質問形式にすることで、人間の認知負荷が下がる。

**3. 具体化のコストは後ろに行くほど高い**

コード実装後に「やっぱり仕様変えたい」は高コストだ。着手前に確認するのが最も安い。このシステムはそのコストを最小化している。

## 実装してみて変わったこと

このシステムを入れて3ヶ月が経った。変化をまとめると：

- **手戻り率**: 体感で60〜70%減
- **仕様のドキュメント化**: 自動で蓄積されるようになった
- **プロジェクトへの理解度**: 質問に答え続けることで自分の頭も整理される

最後の点が予想外の副産物だった。AIに質問されることで、「自分がそこまで考えていなかった」に気づく。仕様の穴を人間が見つけるより、AIに見つけてもらう方が早い。

## CLAUDE.mdのテンプレートを公開する

この記事で紹介した自動具体化ルールの完全版テンプレートは、有料部分に掲載している。

そのまま自分のCLAUDE.mdにコピペして使えるよう、いくつかのプロジェクトタイプ別に調整したバージョンも含めた：

- Webアプリ開発向け
- データ処理・スクリプト向け
- ドキュメント作成向け

それぞれで「何を具体化すべきか」の観点が違う。Webアプリはユーザーと権限の話が多く、データ処理は入出力の形式、ドキュメントは読者と目的が重要になる。

## おわりに

「AIに質問させる」という発想の転換は、Claude Codeの使い方を根本から変えた。

指示する側から、対話する側へ。

AIを道具として使うより、AIを「一緒に仕様を詰める相手」として扱う方が、最終的なアウトプットの質は上がる。

次の記事では、この自動具体化システムをさらに発展させた「AIレビュアー」の仕組みを紹介する予定だ。コードを書かせるだけでなく、書いたコードを別のAIにレビューさせる多段構成の話になる。
