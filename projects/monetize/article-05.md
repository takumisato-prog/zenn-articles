

# Claude Codeで自動具体化システムを作った——曖昧な指示をAIが自動で深掘りする仕組み

## はじめに

「いい感じのLP作っといて」

この一言で、完璧なランディングページが出てきたら最高だろう。でも現実はそうならない。AIに曖昧な指示を投げると、曖昧な成果物が返ってくる。ターゲットも訴求軸もCTAもぼんやりした、誰に向けたのかわからないページが生成される。

「AIが使えない」のではない。**指示が足りていない**のだ。

ただ、問題はそこではない。本当の問題は、**自分の指示が曖昧であることに、自分では気づけない**ということだ。人間は「頭の中にある情報」を無意識に前提として省略する。自分にとっては自明なことを、AIに伝え忘れる。

だから自分は発想を逆転させた。

**人間がAIに質問するのではなく、AIが人間に質問する。**

CLAUDE.mdに「自動具体化ルール」を書き込むことで、曖昧な指示を検知したらClaude Code側から自動的に深掘り質問を飛ばしてくる仕組みを作った。これを導入してから、成果物の一発OKの確率が体感で3倍以上になった。

この記事では、その設計思想と実装方法を全て公開する。

---

## なぜ「AIに質問させる」のか——設計思想の根幹

あなたは、AIに指示を出すとき「もっと具体的に書かなきゃ」と思ったことはないだろうか？

プロンプトエンジニアリングの記事を読むと、必ず「具体的に書きましょう」と書いてある。ターゲットを明記しろ、トーンを指定しろ、文字数を決めろ、参考URLを貼れ——。

正しい。正しいのだが、**それができたら苦労しない**。

自分がクライアントワークをやっていて痛感したのは、「依頼者自身が要件を整理できていないケースが大半」ということだ。「なんかいい感じのサイトが欲しい」「売上が上がる広告を出したい」——これが生の依頼だ。

従来のやり方だと、人間のディレクターが「ターゲットは？」「予算は？」「納期は？」とヒアリングして要件を固める。このヒアリング能力こそが、優秀なディレクターの価値だった。

自分はこのヒアリング能力をClaude Codeに移植した。それが「自動具体化システム」だ。

核となるコンセプトは3つある。

1. **曖昧さを検知する**：指示文に含まれる情報量を自動判定する
2. **質問を生成する**：不足情報を補うための質問を、優先順位をつけて出す
3. **段階的に深掘りする**：一度に大量の質問を浴びせず、1〜2問ずつ聞く

この3つをCLAUDE.mdに書き込むだけで、Claude Codeが「優秀なディレクター」として振る舞い始める。

## 自動具体化ルールの全体設計

実装の話に入る前に、全体のフローを見てほしい。

```
ユーザーの指示入力
       │
       ▼
┌──────────────────────┐
│ 曖昧さ判定エンジン    │
│ （5つの判定条件）     │
└──────────┬───────────┘
           │
     ┌─────┴─────┐
     │           │
  具体的       曖昧
     │           │
     ▼           ▼
  即座に      質問生成
  作業開始    （優先度順）
              │
              ▼
         ユーザー回答
              │
              ▼
         再判定 → 具体的になったら作業開始
                → まだ曖昧なら追加質問（最大3ラウンド）
```

ポイントは「最大3ラウンド」という上限だ。質問が延々と続くとユーザーはストレスを感じる。3往復で要件が固まらなければ、その時点でわかっている情報をベースに仮説を立てて作業を開始し、途中で確認を入れる設計にしている。

## CLAUDE.mdへの実装——コピペで使える具体化ルール

ここからが本題だ。自分がCLAUDE.mdに実際に書いている自動具体化ルールの全文を公開する。

```markdown
## 自動具体化システム（全タスクに常時適用）

### 目的
ユーザーの入力が曖昧な場合、作業着手前に自動で具体化の質問を行い、
成果物の品質と一発OK率を最大化する。

### 曖昧さ判定条件（以下のうち1つでも該当したら「曖昧」と判定）

1. **5W1H欠損**: Who（誰向け）/ What（何を）/ Why（なぜ）/ When（いつまで）/ Where（どこで使う）/ How（どうやって）のうち、2つ以上が不明
2. **数値不在**: 文字数・予算・期限・目標数値など、定量的な指標がゼロ
3. **固有名詞不在**: ターゲット企業名・サービス名・競合名・ツール名など、具体的な名詞がゼロ
4. **曖昧修飾語の検出**: 「いい感じに」「なんとなく」「適当に」「ざっくり」「シンプルに」「かっこよく」などの主観的修飾語が含まれている
5. **ゴール不明**: 成果物の利用シーン・最終的な判断基準が読み取れない

### 質問生成ルール

- 1回の返答につき質問は **最大2問** に絞る
- 最も成果物の品質に影響する曖昧点を優先する
- 質問には **選択肢を3つ付ける**（ユーザーの回答負荷を下げるため）
- 3ラウンド（計6問）で具体化が完了しなかった場合、現時点の情報で仮説を立てて作業開始し、中間確認を挟む

### 質問の優先順位（上ほど優先）

1. ターゲット（誰に届けるのか）
2. ゴール（何を達成したいのか）
3. 制約（予算・納期・NG条件）
4. トーン（フォーマル/カジュアル/専門的）
5. フォーマット（形式・分量・媒体）

### 出力テンプレート

曖昧と判定した場合、以下の形式で質問する：

---
📋 **具体化ヒアリング**（作業前確認）

いただいた内容を確認しました。より良い成果物にするために確認させてください。

**Q1: [最優先の質問]**
  - A) [選択肢1]
  - B) [選択肢2]
  - C) [選択肢3]
  - D) その他（自由記述）

**Q2: [次に重要な質問]**
  - A) [選択肢1]
  - B) [選択肢2]
  - C) [選択肢3]
  - D) その他（自由記述）

上記を教えていただければ、すぐに着手します。
---
```

このルールをCLAUDE.mdに貼るだけで、Claude Codeの振る舞いが劇的に変わる。試しに「LP作って」とだけ入力してみてほしい。Claudeが自動的にターゲットやゴールを聞いてくるはずだ。

## 選択肢付き質問が生んだ意外な効果

あなたは、自由記述とクイズ形式、どちらが答えやすいだろうか？

自分がこのシステムを設計する際、最初は選択肢なしの自由記述で質問させていた。「ターゲットは誰ですか？」とだけ聞く形式だ。

これが失敗だった。

理由は単純で、**自由記述の質問は回答負荷が高い**からだ。「ターゲットは誰ですか？」と聞かれても、依頼者自身がまだ解像度を持っていないことが多い。結果として「30代くらいの人」みたいなぼんやりした回答が返ってきて、具体化が進まない。

選択肢を付けた途端、状況が変わった。

```
Q1: このLPのメインターゲットは？
  A) 個人事業主・フリーランス（年商500万〜3000万）
  B) 中小企業の経営者・役員（従業員10〜50名）
  C) 大企業のDX推進担当者
  D) その他（自由記述）
```

選択肢があると、依頼者の頭の中で「Aに近いけど、もう少し年商が高い層だな」という思考が走る。選択肢が**思考の足場**になるのだ。

しかもこの選択肢自体をClaude Codeが文脈に応じて動的に生成してくれるため、自分がテンプレートを用意する必要はない。CLAUDE.mdには「選択肢を3つ付ける」とだけ書いておけば、あとはClaudeが依頼内容に合った選択肢を自動で考えてくれる。

この仕組みを入れてから、ヒアリング1ラウンド（質問2問）だけで作業着手できるケースが全体の約7割になった。導入前は、チャットで5〜6往復しても要件が固まらないことがザラだったので、劇的な改善だ。

## 実際の動作例——「ブログ記事書いて」と投げてみた

百聞は一見に如かず。実際の動作例を見てもらいたい。

**自分の入力:**
> ブログ記事書いて。Claude Codeについて。

**Claude Codeの応答:**

```
📋 具体化ヒアリング（作業前確認）

Claude Codeについてのブログ記事ですね。
より刺さる記事にするために確認させてください。

Q1: この記事のメインターゲットは？
  A) Claude Codeを初めて使う初心者（インストールから知りたい）
  B) すでに使っているが、もっと活用したい中級者
  C) エンジニア向けの技術的な深掘り記事を求めている層
  D) その他（自由記述）

Q2: 記事のゴール（読者にどうなってほしい？）
  A) Claude Codeを試してみたいと思ってもらう（認知・興味喚起）
  B) 具体的なテクニックを持ち帰ってもらう（実用・ノウハウ）
  C) 自分のサービスや有料記事への導線にする（集客・マネタイズ）
  D) その他（自由記述）
```

**自分の回答:**
> Q1はB、Q2はBで。CLAUDE.mdの活用法を中心に。

**Claude Codeの応答:**

```
了解しました。以下の方針で執筆します。

■ ターゲット: Claude Code中級者（基本操作は知っているが活用が浅い層）
■ ゴール: CLAUDE.mdの具体的テクニックを持ち帰ってもらう
■ 想定媒体: Zenn（技術ブログ）
■ 文字数: 3,000〜4,000字
■ トーン: 体験ベースのカジュアル技術記事

この方針で問題なければ、執筆を開始します。
```

たった1往復で、ここまで要件が固まった。最初の「ブログ記事書いて。Claude Codeについて。」という曖昧な指示が、5W1Hが揃った具体的な要件に変換されている。

## 「3ラウンド上限」と「仮説ベース着手」の設計意図

ヒアリングの質問回数に上限を設けたのには理由がある。あなたは、AIに何度も質問されたらどう感じるだろうか？

自分が最初に自動具体化を試したとき、上限なしで質問させていた。すると5回、6回と質問が続くことがある。これは体験として最悪だ。「聞いてばかりで全然作ってくれないじゃん」とストレスが溜まる。

優秀な人間のディレクターを思い浮かべてほしい。彼らは3往復くらいで要件をまとめ、不明点が残っていても「おそらくこういうことですよね？」と仮説を立てて動き始める。そして中間成果物を見せながら軌道修正する。

これと同じ行動をClaude Codeにさせるために、CLAUDE.mdに以下のルールを追加している。

```markdown
### 仮説ベース着手ルール（3ラウンド経過時に適用）

3ラウンドの質問で具体化が完了しなかった場合：

1. 現時点で判明している情報を箇条書きで整理する
2. 不明点について「最も可能性が高い」仮説を明示する
3. 仮説に基づいて作業を開始する
4. 成果物の30%完成時点で中間レビューを依頼する

中間レビュー時の確認テンプレート：

---
📝 中間確認（方向性チェック）

現在30%まで完成しました。方向性を確認させてください。

【仮説として進めた点】
- [仮説1]: ○○と想定して進めています
- [仮説2]: △△と想定して進めています

【確認したいこと】
- 上記の仮説で問題ないか
- 方向修正があれば今のタイミングで教えてください

このまま進めてよければ「OK」とだけ返してください。
---
```

「OK」の一言で続行できる設計にしているのもポイントだ。中間確認のたびに長文を書かせると、ユーザーの負荷が高くなる。確認だけなら一言で済むようにする——この「回答負荷の最小化」が、自動具体化システム全体を貫く設計思想だ。

## 導入前後で何が変わったか——定量的な変化

自分の環境で、自動具体化システム導入前後の変化を記録していた。サンプル数は約60タスク（導入前30、導入後30）での比較だ。

| 指標 | 導入前 | 導入後 | 変化 |
|------|--------|--------|------|
| 要件確定までの往復数 | 平均5.2回 | 平均1.8回 | **65%減** |
| 成果物の一発OK率 | 23% | 67% | **約3倍** |
| 手戻り（やり直し）回数 | 平均2.1回 | 平均0.6回 | **71%減** |
| 1タスクの完了時間 | 平均38分 | 平均15分 | **60%減** |

特に「一発OK率」の変化が大きい。曖昧な指示のまま作業に入ると、出来上がったものが期待と違うので手戻りが発生する。自動具体化を挟むことで、作業開始前に認識のズレを潰せる。結果として「作ったけどやり直し」という無駄が激減した。

## よくある質問と注意点

**Q: 毎回質問されるのがウザくならないか？**

指示が十分に具体的であれば、質問はスキップされる。「5W1Hのうち2つ以上が不明」という条件を入れているので、具体的な指示には即座に作業開始する。曖昧なときだけ発動する仕組みだ。

**Q: 質問の精度がイマイチなときは？**

CLAUDE.mdの「質問の優先順位」を自分の業務に合わせてカスタマイズすればいい。例えばデザイン業務が多いなら「参考デザイン・ブランドガイドラインの有無」を優先順位の上位に入れる。

**Q: 他のAIツールでも使える？**

考え方自体は汎用的だ。ただし、CLAUDE.mdのような「自動読み込みされる設定ファイル」が使えるのはClaude Codeの強みである。他ツールではシステムプロンプトに同等の内容を書くことになるが、プロジェクトごとの切り替えが手間になる。

## まとめ——AIの質を上げるのは「指示力」ではなく「仕組み」

自分がこのシステムを作って確信したのは、**AIの出力品質を決めるのは「プロンプトの上手さ」ではなく「入力を具体化する仕組みがあるかどうか」**だということだ。

毎回完璧なプロンプトを書ける人間はいない。疲れているとき、急いでいるとき、考えがまとまっていないとき——そういう「人間のコンディションが悪いとき」こそ、AIが補完してくれる仕組みが必要だ。

自動具体化システムは、人間の弱さをAIの仕組みで補うアプローチである。「AIに質問させる」という逆転の発想が、結果として人間とAIの協働を最もスムーズにする。

CLAUDE.mdにたった数十行のルールを書くだけで、あなたのClaude Codeは「言われたことをやるだけのツール」から「一緒に考えてくれるパートナー」に変わる。
