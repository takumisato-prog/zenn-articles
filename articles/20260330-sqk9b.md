## この記事の前提

私はClaude Codeで47名のAIエージェントが稼働する会社「Altus」を運営している。スキルファイルの基本的な作り方や、個別の専門家AIの構築方法については過去記事で書いた。

今回のテーマは「その先」だ。個々のエージェントは優秀でも、**エージェント同士が連携できなければ組織として機能しない**。実際に直面した「部門間の情報断絶」「タスクの委譲失敗」「責任の曖昧化」という3つの地獄と、それをスキルファイルの設計で解決した方法を書く。

## 単体エージェントの限界に気づいた瞬間

Altusの初期構成では、各エージェントが独立して動いていた。マーケティング担当のエージェントがLP案を作り、デザイン担当が実装し、QA担当がチェックする。一見うまく回りそうに見えた。

しかし現実はこうなった。

- マーケ担当が「CTAボタンは赤」と指示したのに、デザイン担当はブランドガイドラインに従って青にした
- QA担当はデザインの意図を知らないので「仕様通り」と判定した
- 最終成果物を見た私は「誰がこの判断をした？」と混乱した

**個々のスキルファイルをどれだけ磨いても、エージェント間のインターフェースが設計されていなければ組織は破綻する。**これが最初の学びだった。

## 部門間連携の設計パターン：3層アーキテクチャ

試行錯誤の末にたどり着いたのが、スキルファイル内に「連携プロトコル」を明示的に埋め込む設計だ。

### 第1層：コンテキスト共有ファイル

各部門のエージェントが読み書きする共有ファイルを定義する。

```markdown
# skills/marketing/lp-planning.md

## 連携プロトコル

### 出力先（必須）
タスク完了時、以下のファイルに成果物のサマリを書き出すこと：
- `docs/handoffs/marketing-to-design.md`

### サマリに含める必須項目
1. 今回の施策の目的（KPI含む）
2. 譲れない要件（色・配置・文言で確定しているもの）
3. デザイン側に委ねる領域（明示的に「任せる」と書く）
4. 判断に迷った場合のエスカレーション先：CEO（人間）

### 入力元（参照必須）
作業開始前に必ず以下を読むこと：
- `docs/brand-guidelines.md`
- `docs/handoffs/design-to-marketing.md`（前回のフィードバック）
```

ポイントは**「任せる領域」の明示**だ。人間の組織でも同じだが、「ここは自分の管轄外」と認識できないエージェントは、すべてを自分で決めようとするか、何も決められなくなる。

### 第2層：ハンドオフファイルの構造化

`docs/handoffs/` ディレクトリに置くファイルには厳密なフォーマットを定めた。

```markdown
# Marketing → Design ハンドオフ

## メタ情報
- 日時: 2025-01-15
- 担当エージェント: marketing-lead
- 優先度: HIGH
- ステータス: READY_FOR_PICKUP

## 施策概要
新規LP「エンタープライズプラン訴求」

## 確定事項（変更不可）
- ヘッドライン: 「導入3日で成果が出る、AI組織構築」
- CTAテキスト: 「無料で試す」
- ファーストビューに導入企業ロゴを配置

## 委譲事項（デザイン判断に委ねる）
- 配色（ブランドガイドライン準拠の範囲で自由）
- レイアウト構成
- アニメーションの有無

## 制約
- ページ読み込み3秒以内
- モバイルファーストで設計
```

`READY_FOR_PICKUP`というステータスが重要だ。デザイン担当のスキルファイルには「ステータスがREADY_FOR_PICKUPのハンドオフファイルのみ作業対象とする」と書いてある。これにより、作成途中のファイルを拾ってしまう事故を防げる。

### 第3層：エスカレーションルール

各スキルファイルに「判断できない場合のルール」を明記する。

```markdown
## エスカレーションポリシー

### 自分で判断してよい範囲
- ブランドガイドラインに明記されている事項
- 過去の類似案件で確立されたパターン

### サブエージェントに委譲する範囲
- コードの実装詳細 → `skills/engineering/frontend-impl.md` を持つエージェントに委譲
- コピーライティングの微調整 → `skills/content/copywriting.md` を持つエージェントに委譲

### 人間にエスカレーションする範囲
- ブランドガイドラインに記載がない新しい判断
- 予算に影響する意思決定
- 2つの確定事項が矛盾する場合
```

## サブエージェント委譲の実装

Claude Codeの `Subagent` を活用する場面で最も効果があったのは、**1つのスキルファイル内で「自分の仕事」と「委譲する仕事」を明確に分離した**ことだ。

```markdown
# skills/design/lp-design-lead.md

## あなたの役割
LP全体のデザインディレクション。構成・トンマナ・UXフローを決定する。

## サブエージェント委譲ルール

### コンポーネント実装の委譲
以下の手順でサブエージェントにタスクを渡すこと：

1. 自分がワイヤーフレーム（テキストベース）を作成する
2. 各セクションごとに実装指示書を作成する
3. サブエージェントに渡す際、以下を必ず含める：
   - セクション名
   - 期待するHTML/CSS構造の概要
   - 使用可能なCSSフレームワーク: Tailwind CSS
   - 参照すべきデザイントークン: `docs/design-tokens.json`
4. サブエージェントの出力を自分がレビューし、全体整合性を確認する

### 委譲してはならないこと
- ページ全体の構成決定
- ナビゲーション設計
- CTAの配置判断

これらは自分の責任範囲であり、サブエージェントに丸投げしない。
```

この設計の肝は「**委譲してはならないこと**」の明記だ。これがないと、リードエージェントがすべてをサブエージェントに流してしまい、誰も全体を見ていない状態になる。Altusで実際に起きた事故だ。

## 47体を統制する「組織CLAUDE.md」の実態

プロジェクトルートの `CLAUDE.md` には、個別のスキルではなく**組織全体のルール**を書いている。抜粋するとこうなる。

```markdown
## 組織運営ルール

### ファイル所有権
- 各ファイルには「オーナーエージェント」が存在する
- 他エージェントのオーナーファイルを直接編集してはならない
- 変更が必要な場合、ハンドオフファイル経由で依頼する

### コンフリクト解決
- 2つのエージェントの指示が矛盾した場合、上位のハンドオフファイルを優先する
- 優先順位: CEO指示 > 部門長ハンドオフ > 担当者判断

### 作業ログ
- 全エージェントは作業完了時に `logs/YYYY-MM-DD/` 配下にログを残す
- ログには「何を判断したか」「なぜその判断をしたか」を必ず含める
```

このログ設計が地味に効いている。問題が起きたとき「なぜこうなった？」をログから追跡できる。47体もいると、デバッグなしでは運用不能だ。

## 設計で得た3つの原則

半年の運用で確信した原則を3つにまとめる。

**1. 「何を任せるか」より「何を任せないか」を書け**
エージェントは指示がなければ最大限の自律性を発揮しようとする。境界を引くのは「やるな」の指示だ。

**2. ハンドオフは非同期前提で設計せよ**
エージェント間でリアルタイム通信を期待するとカオスになる。ファイルベースの非同期通信が最も安定する。

**3. 判断ログは贅沢品ではなく必需品である**
「何をやったか」だけでなく「なぜそう判断したか」のログがないと、スキルファイルの改善サイクルが回らない。

## まとめ

スキルファイルの設計は、単体の「プロンプトエンジニアリング」ではない。47体のエージェントを動かしてわかったのは、**これは組織設計そのものだ**ということだ。人間の組織で「役割定義」「権限委譲」「報告ライン」「エスカレーションルール」が必要なように、AIエージェント組織でもまったく同じ構造が必要になる。

違いがあるとすれば、人間は曖昧なルールでもなんとなく空気を読んで動けるが、AIは書かれていないことを「やらなくていい」と解釈する点だ。だからこそ、スキルファイルの設計精度がそのまま組織の性能になる。
