## はじめに：なぜプレスリリースをAIで自動化するのか

私はClaude Codeで47名のAIエージェントが稼働する会社「Altus」を運営している。この規模になると、新サービスのリリース、提携発表、実績報告など、プレスリリースを出す機会が月に何本もある。

従来のフローはこうだった。

1. 担当者がドラフトを書く（1〜2時間）
2. 社内レビュー（半日〜1日）
3. PR TIMESの管理画面にログインして手動入稿（30分）
4. 画像の設定やカテゴリ選択（15分）

合計で最低半日、長ければ2日かかる作業だ。これをClaude CodeとPR TIMES APIで自動化したところ、**素材を渡してから投稿準備完了まで約3分**になった。本記事ではその具体的な方法を解説する。

## 全体アーキテクチャ

```
[素材ファイル群] → [Claude Code] → [プレスリリースJSON生成] → [PR TIMES API] → [下書き保存]
     │                    │
     │                    ├── CLAUDE.md（会社のトンマナ定義）
     │                    └── スキルファイル（PR専門エージェント）
     │
     ├── product_info.md（サービス概要）
     ├── quotes.md（コメント素材）
     └── images/（メイン画像・サブ画像）
```

ポイントは、**Claude Codeが「原稿生成」と「API投稿」の両方を一気通貫で実行する**点だ。別々のツールを繋ぐ必要はない。

## ステップ1：PR TIMES APIの準備

PR TIMESは法人向けにAPIを提供している。管理画面の「設定」→「API設定」からAPIキーを取得する。

```bash
# .envファイルに保存
PRTIMES_API_KEY=your_api_key_here
PRTIMES_COMPANY_ID=your_company_id
```

APIのエンドポイントは以下の通りだ。

```
POST https://hackathon.prtimes.com/api/v1/press_releases
```

※ PR TIMESのAPIは公式ドキュメントでエンドポイントが案内されている。契約プランによって利用可否が異なるため、事前に確認が必要だ。ここではHackathon APIを例にしているが、本番環境では正式なAPIエンドポイントを使用する。

## ステップ2：プレスリリース生成用のスキルファイル

Claude Codeのスキル機能を使い、PR専門エージェントとしてのルールを定義する。

```markdown
<!-- .claude/skills/press_release_writer.md -->

# プレスリリース作成スキル

## 役割
あなたはAltus社のPR担当エージェントである。

## 構成ルール
1. タイトル：30文字以内。数字を1つ以上含める
2. サブタイトル：60文字以内で補足
3. リード文：5W1Hを最初の3文で網羅する
4. 本文構成：
   - 背景・課題（なぜ今これを出すのか）
   - サービス/発表の詳細
   - 代表者または関係者コメント（引用形式）
   - 今後の展望
   - 会社概要ボイラープレート
5. 本文の文字数：800〜1200字

## トンマナ
- 「です・ます」調で統一
- 誇張表現禁止（「革命的」「世界初」等は根拠がない限り使わない）
- 具体的な数字・実績を積極的に盛り込む

## 出力形式
JSON形式で出力すること。キー名はPR TIMES APIのパラメータに準拠する。
```

## ステップ4：Claude Codeで生成からAPI投稿まで実行

ここが核心部分だ。Claude Codeに以下のように指示する。

```bash
claude "input/ディレクトリの素材を読み込み、.claude/skills/press_release_writer.mdのルールに従って
プレスリリースを生成し、PR TIMES APIに下書きとして投稿するスクリプトを作成・実行してくれ"
```

Claude Codeが生成する投稿スクリプトは概ね以下のような構造になる。

```python
#!/usr/bin/env python3
"""PR TIMES API への下書き投稿スクリプト"""

import os
import json
import requests
from dotenv import load_dotenv

load_dotenv()

API_KEY = os.getenv("PRTIMES_API_KEY")
COMPANY_ID = os.getenv("PRTIMES_COMPANY_ID")
BASE_URL = "https://hackathon.prtimes.com/api/v1"

def create_press_release(content: dict) -> dict:
    """プレスリリースを下書きとして作成"""
    headers = {
        "Authorization": f"Bearer {API_KEY}",
        "Content-Type": "application/json"
    }

    payload = {
        "title": content["title"],
        "sub_title": content["sub_title"],
        "body": content["body"],
        "release_type": "new_service",
        "status": "draft",  # 必ず下書きで保存
        "company_id": COMPANY_ID
    }

    response = requests.post(
        f"{BASE_URL}/press_releases",
        headers=headers,
        json=payload
    )

    if response.status_code == 201:
        print(f"✅ 下書き保存完了: {response.json().get('id')}")
        return response.json()
    else:
        print(f"❌ エラー: {response.status_code}")
        print(response.text)
        raise Exception(f"API Error: {response.status_code}")

if __name__ == "__main__":
    # Claude Codeが生成したプレスリリースJSONを読み込み
    with open("output/press_release.json", "r") as f:
        content = json.load(f)

    result = create_press_release(content)
    print(json.dumps(result, ensure_ascii=False, indent=2))
```

**重要な設計判断として、`status`は必ず`"draft"`にしている。** 自動で公開まではしない。最終確認は人間が行う。これは「自動化の範囲」を意図的に制限するためだ。

## ステップ5：生成されたプレスリリースの品質チェック

Claude Codeは生成後に自動でセルフチェックも行える。私は以下のようなバリデーションを組み込んでいる。

```python
def validate_press_release(content: dict) -> list[str]:
    """プレスリリースの品質チェック"""
    errors = []

    # タイトル文字数
    if len(content["title"]) > 30:
        errors.append(f"タイトルが{len(content['title'])}文字（上限30文字）")

    # 本文文字数
    body_len = len(content["body"])
    if body_len < 800 or body_len > 1200:
        errors.append(f"本文が{body_len}文字（800〜1200文字の範囲外）")

    # 禁止ワードチェック
    banned_words = ["革命的", "世界初", "業界初", "画期的"]
    for word in banned_words:
        if word in content["body"]:
            errors.append(f"禁止ワード検出: {word}")

    # 数字の含有チェック（具体性の担保）
    import re
    if not re.search(r'\d+', content["body"]):
        errors.append("本文に具体的な数字が含まれていない")

    return errors
```

バリデーションでエラーが出た場合、Claude Codeがその場で修正して再生成する。この「生成→検証→修正」のループがClaude Codeの内部で完結するのが強い。

## 実運用で気づいた3つのこと

### 1. ボイラープレートはCLAUDE.mdに書くべき

会社概要の定型文はスキルファイルではなくCLAUDE.mdに記載している。プレスリリース以外の場面（提案書、メール等）でも参照されるためだ。

### 2. 画像のアップロードは別ステップにした方が安定する

PR TIMES APIへの画像アップロードは、テキスト投稿とは別のリクエストとして分離した。マルチパートでの一括送信は失敗率が高かった。

### 3. 「下書き→人間確認→公開」のフローは崩さない

完全自動公開の誘惑はある。だが、プレスリリースは会社の公式発表だ。事実誤認が一つでもあれば信用に関わる。**AIに任せるのは「叩き台を最高品質で作る」ところまで**。公開ボタンは人間が押す。この線引きが、47エージェント運用で学んだ最大の教訓だ。

## まとめ

| 工程 | 従来 | 自動化後 |
|------|------|----------|
| ドラフト作成 | 1〜2時間 | 30秒 |
| レビュー・修正 | 半日 | 2分（自動バリデーション+手動最終確認） |
| PR TIMES入稿 | 30分 | 自動 |
| **合計** | **半日〜2日** | **約3分+最終確認5分** |

Claude Codeの本質的な強みは、「文章生成」と「コード実行」がシームレスに繋がる点だ。原稿を作り、バリデーションし、APIを叩く。この一連の流れが一つのセッション内で完結する。

プレスリリースに限らず、「定型フォーマット×外部API」の組み合わせは他にも応用できる。求人原稿の自動投稿、SNS投稿の一括生成、レポートの定期配信——Claude Codeはこうした実務の自動化に最も適したツールだと確信している。
