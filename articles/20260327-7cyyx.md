## はじめに──AIエージェントは「呼ばれたら動く」だけでは足りない

47名のAIエージェントで構成される会社「Altus」を運営して数ヶ月。最初に直面した壁は「誰がAIに指示を出すのか」だった。

人間が毎朝「レポート作って」「ログ見て」「Slackに投稿して」と指示する運用は、AIが47名いても結局ボトルネックは人間1人だ。これでは意味がない。

そこで導入したのが、Claude Codeのスケジュール実行──つまり**cron等のスケジューラからClaude Codeを定期的にキックする仕組み**である。この記事では、実際にAltusで動かしている定期実行タスクの設計パターンと実装方法を具体的に解説する。

## Claude Codeを「定期実行」する基本構成

Claude Codeはターミナルで動くCLIツールだ。つまり、**任意のスケジューラからコマンドとして呼び出せる**。特別なAPIキーの設定やWebhookは不要で、`claude`コマンドをそのまま叩くだけでいい。

基本形はこうなる。

```bash
# 最もシンプルな定期実行
claude -p "今日のタスクリストを確認し、期限切れのものをSlackに報告して"
```

`-p`（`--print`）フラグを使うと、対話モードに入らずプロンプトを渡して結果を標準出力に返す。これがスケジュール実行の要になる。

さらに実用上重要なのが以下のオプションだ。

```bash
claude -p "プロンプト" \
  --model claude-sonnet-4-20250514 \
  --max-turns 10 \
  --output-format json
```

- `--max-turns`: エージェントがツールを使って何往復まで自律的に動くかを制限する。暴走防止に必須
- `--output-format json`: 後続処理でパースしやすくなる

## crontabでの設定例

Altusで実際に動いている定期タスクの一部を紹介する。

```crontab
# 毎朝9時: 日次レポート生成
0 9 * * * cd /home/altus/workspace && /usr/local/bin/claude -p "$(cat prompts/daily-report.md)" --max-turns 15 >> logs/daily-report.log 2>&1

# 毎時0分: エラーログ監視
0 * * * * cd /home/altus/workspace && /usr/local/bin/claude -p "$(cat prompts/error-watch.md)" --max-turns 5 --output-format json | /usr/local/bin/notify-slack.sh

# 毎週月曜10時: 週次コードレビュー
0 10 * * 1 cd /home/altus/workspace && /usr/local/bin/claude -p "$(cat prompts/weekly-review.md)" --max-turns 20 >> logs/weekly-review.log 2>&1
```

ポイントは3つある。

1. **プロンプトはファイルに外出しする**。crontab内にプロンプトを直書きすると、エスケープ地獄になる
2. **必ず`cd`でワーキングディレクトリを指定する**。CLAUDE.mdの読み込みやファイルアクセスのパスが狂う
3. **`claude`のフルパスを指定する**。cronはPATHが最小限なので、`which claude`で確認したパスを使う

## プロンプトファイルの設計──「指示書」を渡す感覚

外出ししたプロンプトファイルの実例を1つ紹介する。これはAltusの「監査担当エージェント」が毎時実行するエラー監視用だ。

```markdown
# prompts/error-watch.md

## タスク
直近1時間のアプリケーションログを確認し、異常を検知した場合はレポートを生成せよ。

## 手順
1. `logs/app/` 配下の最新ログファイルを読み込む
2. ERROR または FATAL レベルのログを抽出する
3. 同一エラーはグルーピングし、発生回数を集計する
4. 以下のフォーマットでJSONを出力する

## 出力フォーマット
{
  "timestamp": "ISO8601",
  "has_errors": true/false,
  "summary": "概要テキスト",
  "errors": [
    {
      "level": "ERROR",
      "message": "エラー内容",
      "count": 3,
      "first_seen": "ISO8601",
      "file": "ファイルパス"
    }
  ]
}

## 制約
- ログファイルが存在しない場合は has_errors: false で即座に返す
- ファイルの書き込み・削除は行わない（読み取りのみ）
```

ここで重要なのは**「制約」セクション**だ。定期実行ではヒューマンレビューが入らないため、「やっていいこと」「やってはいけないこと」を明示的に書く。これを怠ると、深夜3時にエージェントが本番ファイルを消すような事故が起きかねない。

## 実用パターン4選──Altusで実際に回しているもの

### パターン1: Gitリポジトリの自動整理

```bash
# 毎日深夜2時: マージ済みブランチの一覧化
claude -p "$(cat prompts/branch-cleanup.md)" --max-turns 5 \
  --allowedTools "Bash(git branch:*),Bash(git log:*)" \
  >> logs/branch-cleanup.log 2>&1
```

`--allowedTools`で使用可能なツールを限定している。`git push`や`git branch -d`を許可しないことで、**リスト化はするが削除はしない**という安全な運用にしている。

### パターン2: ドキュメントの鮮度チェック

```bash
# 毎週水曜: READMEとコードの乖離チェック
claude -p "README.mdの記述と実際のコードを比較し、乖離している箇所をリストアップせよ" \
  --max-turns 15
```

これは地味だが非常に役立つ。ドキュメントが腐る問題をAIが週次で検知してくれる。

### パターン3: 依存パッケージの脆弱性サマリ

```bash
# 毎日朝7時: npm auditの結果を要約
npm audit --json 2>/dev/null | claude -p "標準入力のnpm audit結果を読み、critical/highの脆弱性だけ日本語で要約せよ" \
  --max-turns 3
```

パイプで前段コマンドの出力を食わせるパターン。Claude Codeは標準入力も受け取れるので、既存のCLIツールとの連携が非常に簡単だ。

### パターン4: 日次スタンドアップレポート

```bash
# 毎朝8:50: 昨日のGitコミットからスタンドアップ用サマリ生成
claude -p "$(cat prompts/standup.md)" --max-turns 10 --output-format json \
  | jq -r '.result' | post-to-slack.sh "#daily-standup"
```

`--output-format json`で返ってくるJSONの`.result`フィールドにテキスト出力が入るので、`jq`で取り出してSlackに流している。

## 実運用で学んだ注意点

### 1. 並列実行の制御

cronで複数タスクが同時刻に走ると、API レート制限にぶつかる。Altusでは**flockコマンドでロックファイルを使い、直列化**している。

```bash
flock -n /tmp/claude-scheduler.lock claude -p "..."
```

### 2. コスト監視は必須

定期実行はトークンが静かに積み上がる。Altusでは週次でClaude APIの使用量を集計するタスク自体もスケジュール実行している（メタ的だが実用的だ）。

### 3. CLAUDE.mdとの組み合わせ

`cd`でワーキングディレクトリを正しく指定すれば、そのディレクトリのCLAUDE.mdが自動で読み込まれる。つまり**タスクごとにディレクトリを分け、それぞれにCLAUDE.mdを置く**ことで、エージェントの振る舞いをタスク単位で制御できる。

```
/home/altus/workspace/
├── error-watch/
│   ├── CLAUDE.md          # 監視エージェント用設定
│   └── prompts/
├── code-review/
│   ├── CLAUDE.md          # レビューエージェント用設定
│   └── prompts/
└── reporting/
    ├── CLAUDE.md          # レポートエージェント用設定
    └── prompts/
```

この構成にすると、各cronジョブは対応するディレクトリに`cd`するだけで、適切な人格と制約を持ったエージェントとして動作する。

## まとめ

Claude Codeのスケジュール実行は、特別な仕組みではない。CLIツールとしてのClaude Codeを、cronという枯れた技術で定期的に叩く。それだけだ。

だが、この「それだけ」が組織運営を根本から変える。47名のAIエージェントが人間の指示を待たず、決まったタイミングで自律的に仕事をする。人間は結果を確認するだけになる。

重要なのは以下の3点だ。

- **プロンプトはファイルに外出しし、「制約」を必ず明記する**
- **`--allowedTools`と`--max-turns`で行動範囲を制限する**
- **タスクごとにディレクトリを分け、CLAUDE.mdで人格を分離する**

AIエージェントを「呼んだら動く便利ツール」から「放っておいても仕事する同僚」に変えるのが、スケジュール実行の本質である。
