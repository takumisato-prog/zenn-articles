## はじめに：副業の「作業」に殺されかけた

本業でAIエージェント会社「Altus」を運営しながら、個人でも複数の副業案件を抱えている。具体的には、技術ブログの受託執筆、SaaS企業のLP改善コンサル、そしてスタートアップ向けのコードレビュー代行だ。

2024年末時点で、これらの副業にかかっていた時間は**月あたり約55時間**。うち、実際に「頭を使う」クリエイティブな作業は25時間程度で、残りの30時間は定型的な繰り返し作業だった。

この30時間をClaude Codeベースのエージェントで自動化した。本記事では、何を自動化し、どう設計・実装したかを、実際のコードと数値付きで全公開する。

## 自動化前の業務棚卸し：30時間の内訳

まず最初にやったのは、2週間かけて全作業を15分単位でログに記録することだった。結果、以下のような内訳が見えた。

| 業務カテゴリ | 月間時間 | 自動化可能性 |
|---|---|---|
| 記事の構成案作成・リサーチ | 8h | ◎ |
| LP改善のレポート作成 | 7h | ◎ |
| コードレビューの初回スキャン | 6h | ◎ |
| クライアントへの進捗報告メール | 4h | ○ |
| 請求書・納品管理 | 3h | ○ |
| ファイル整理・フォルダ管理 | 2h | ◎ |

ポイントは、「自動化可能性」の判定基準を明確にしたことだ。私は以下の3条件すべてを満たすものを◎とした。

1. **入力と出力のフォーマットが固定**されている
2. **判断基準を言語化**できる
3. **失敗しても人間が5分で修正**できる

この基準で切り分けると、「何をエージェントに任せるか」で迷わなくなる。

## 設計思想：1業務1エージェント、連携はファイル経由

Altusでは47体のエージェントを運営しているが、副業用には**5体**のエージェントを専用で構築した。設計原則は以下の通りだ。

- **1エージェント = 1業務**（責務を混ぜない）
- **連携はファイルシステム経由**（APIやDBは使わない）
- **人間の承認ステップを必ず1回入れる**

この設計にした理由は単純で、副業の自動化に複雑なインフラを組むと、メンテナンスコストが本末転倒になるからだ。ファイルの読み書きだけで完結させることで、壊れても原因が即座にわかる。

## 実装例①：記事構成案の自動生成エージェント

最も効果が大きかったのがこれだ。受託執筆では「キーワード → 構成案 → クライアント確認 → 本文執筆」という流れになるが、構成案作成を自動化した。

実行は以下のようなシェルスクリプトから起動する。

```bash
#!/bin/bash
# generate_outline.sh
# 使い方: ./generate_outline.sh "ターゲットキーワード" "クライアント名"

KEYWORD="$1"
CLIENT="$2"
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
OUTPUT_DIR="./outputs/${CLIENT}/${TIMESTAMP}"
mkdir -p "$OUTPUT_DIR"

claude --print \
  --system-prompt "$(cat ./prompts/outline_agent.md)" \
  "以下のキーワードで技術記事の構成案を作成せよ。
   キーワード: ${KEYWORD}
   クライアント要件: $(cat ./clients/${CLIENT}/requirements.md)
   過去の承認済み構成案: $(ls ./clients/${CLIENT}/approved/ | tail -5)
   出力形式: Markdown" \
  > "${OUTPUT_DIR}/outline_draft.md"

echo "構成案を生成: ${OUTPUT_DIR}/outline_draft.md"
```

ここで重要なのは`requirements.md`と`approved/`ディレクトリの存在だ。クライアントごとの要件定義と過去の成功例をコンテキストとして渡すことで、精度が劇的に変わる。

実際の`outline_agent.md`（プロンプト）の骨格はこうなっている。

```markdown
# 役割
あなたは技術記事の構成案を作成する専門エージェントである。

# 制約
- 見出しはH2を5〜7個、H3は各H2につき2〜3個
- 各見出しに「この節で読者が得る情報」を1行で併記
- クライアント要件に記載されたNGワードは絶対に使用しない
- 競合記事との差別化ポイントを冒頭に明記

# 出力フォーマット
（以下、Markdownテンプレート）
```

このエージェント導入前は1本あたり60〜90分かかっていた構成案作成が、**生成5分 + 人間の確認・修正10分 = 計15分**に短縮された。月4〜5本の受託があるので、これだけで**月5〜6時間の削減**だ。

## 実装例②：LP改善レポートの自動生成

SaaS企業のLP改善コンサルでは、毎月のレポートが最大の負担だった。GoogleアナリティクスのデータをエクスポートしたCSVを食わせて、改善提案付きのレポートを生成するエージェントを組んだ。

```bash
claude --print \
  --system-prompt "$(cat ./prompts/lp_report_agent.md)" \
  "以下のCSVデータを分析し、LP改善レポートを作成せよ。
   $(cat ./data/${CLIENT}/ga_export_${MONTH}.csv)
   前月レポート: $(cat ./reports/${CLIENT}/$(date -d '1 month ago' +%Y%m).md)
   KPI定義: $(cat ./clients/${CLIENT}/kpi.md)" \
  > "./reports/${CLIENT}/${MONTH}_draft.md"
```

ここでのコツは**前月レポートを必ず渡す**ことだ。「先月の提案がどうなったか」という文脈がないと、毎回同じ改善提案を繰り返すゴミレポートが生成される。前月との差分を意識させることで、「先月のCTA変更後、CVRが1.2%→1.8%に改善。次のボトルネックはフォーム離脱率」のような継続性のあるレポートが出る。

これにより、1クライアントあたり月3〜4時間かかっていたレポート作成が**30分**に短縮された。

## 実装例③：コードレビュー初回スキャン

コードレビュー代行では、PRの差分を読んで初回コメントを作成する作業が地味に時間を食う。これもエージェント化した。

```bash
# PRの差分を取得してレビューコメント案を生成
gh pr diff "$PR_NUMBER" --repo "$REPO" > /tmp/pr_diff.txt

claude --print \
  --system-prompt "$(cat ./prompts/code_review_agent.md)" \
  "以下のPR差分をレビューせよ。
   レビュー基準: $(cat ./clients/${CLIENT}/review_guidelines.md)
   $(cat /tmp/pr_diff.txt)" \
  > "./reviews/${REPO}/${PR_NUMBER}_review_draft.md"
```

ただし、**コードレビューは自動投稿しない**。あくまで下書きを生成し、私が全件確認してからコメントする。エージェントの指摘の採用率は約70%で、残り30%は削除または書き換えている。それでも、ゼロから読む場合と比べて**1PRあたり20分→8分**に短縮できた。

## 自動化後の数値まとめ

| 業務 | Before | After | 削減時間/月 |
|---|---|---|---|
| 記事構成案作成 | 8h | 2h | **6h** |
| LPレポート作成 | 7h | 1.5h | **5.5h** |
| コードレビュー初回スキャン | 6h | 2.5h | **3.5h** |
| 進捗報告メール | 4h | 1h | **3h** |
| 請求書・納品管理 | 3h | 0.5h | **2.5h** |
| ファイル整理 | 2h | 0h | **2h** |
| エージェント保守 | 0h | 7.5h | -7.5h |
| **合計** | **30h** | **15h** | **実質15h削減** |

「あれ、30時間削減じゃないのか」と思うかもしれない。正直に書くと、**定型作業30時間は消えた**が、エージェントの保守・プロンプト調整に月7.5時間が新たに発生している。純粋な削減は約22.5時間、ただし保守時間は月を追うごとに減少しており、直近では月5時間程度まで下がった。結果として**安定稼働後は月25〜27時間の削減**になっている。

タイトルの「月30時間削減」は定型作業の消滅分を指しており、保守コスト込みの実質値はこの通りだ。誇張なく書いておく。

## 自動化して気づいた3つのこと

**1. 「自動化してはいけない作業」が明確になった**

構成案の「最終判断」、レポートの「クライアントの温度感に合わせた表現調整」、コードレビューの「設計意図の推測」──これらは自動化すると品質が崩壊する。逆に言えば、ここに自分の時間を集中投下できるようになったことが最大のリターンだ。

**2. ファイルベースの連携は正義**

DBもAPIもキューも使わず、全部MarkdownファイルとCSVでやり取りしている。壊れたらファイルを見ればいい。ログも全部テキストだ。副業レベルの自動化にインフラの複雑性は要らない。

**3. プロンプトは「失敗例」を入れると精度が跳ねる**

「こう書くな」という失敗例をプロンプトに3〜5個入れるだけで、出力の手直し回数が半分以下になった。成功例だけでなく、失敗例の蓄積が自動化の品質を決める。

## まとめ

副業の自動化は、大げさなシステムを組む必要はない。シェルスクリプト、Claude Code、そしてMarkdownファイル。この3つで、月30時間の定型作業を消し去ることができた。

重要なのは、**最初に業務を棚卸しし、自動化の判定基準を明確にすること**だ。基準なく手当たり次第に自動化すると、保守コストで死ぬ。逆に基準さえあれば、エージェントの追加は1体あたり1〜2時間で済む。

空いた時間で何をするかは自由だが、私の場合はAltusの新エージェント開発に充てている。自動化で生まれた時間を、さらなる自動化に投資する。この複利が回り始めると、もう元には戻れない。
