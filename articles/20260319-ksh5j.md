## AIだけで動く会社の構造

自分が作ったのは、こういう組織だ。

```
                ┌─────────────────┐
                │   PM（全体統括）  │
                └────────┬────────┘
                         │ 案件振り分け
    ┌────────────────────┼────────────────────┐
    │            │               │             │
    ▼            ▼               ▼             ▼
┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
│ DX支援   │ │  EC・広告 │ │クリエイ  │ │ PR・SNS  │
│   部門   │ │   部門   │ │ ティブ  │ │   部門   │
└──────────┘ └──────────┘ │   部門   │ └──────────┘
                          └──────────┘
    └────────────────────┬────────────────────┘
                         │
                    ┌────▼────┐
                    │  開発   │
                    │  部門   │
                    └─────────┘
```

5つの専門部門と、それを統括するPM。各部門はそれぞれ専門領域を持っている。

| 部門 | 担当領域 |
|------|---------|
| DX支援 | Googleワークスペース導入・業務自動化 |
| EC・広告 | Amazon販売支援・広告運用 |
| クリエイティブ | LP作成・コピーライティング・商品画像 |
| PR・SNS | プレスリリース・SNS運用 |
| 開発 | Webアプリ・自動化スクリプト |
| PM | 案件振り分け・進捗管理 |

これ、全部AIが担っている。

---

## 仕組みはシンプルだった

複雑そうに見えるが、技術的なカラクリはシンプルだ。

**Claude Codeには「CLAUDE.md」というファイルがある。**

プロジェクトのルートに置くと、Claudeが自動的に読み込む設定ファイルだ。ここに「あなたはこういう会社の、こういう役割のAIです」と書くだけで、Claudeの振る舞いが変わる。

```markdown
# AI会社 — 会社定義

## ビジョン
AIを活用して、中小企業・個人事業主の「売上・業務効率・情報発信」を
一気通貫で支援する会社。

## 部門構成
...（各部門の定義）
```

そして「スキルファイル」という仕組みを使うと、専門エージェントを呼び出せる。

---

## スキルファイルとは何か

`.claude/skills/` というフォルダに `.md` ファイルを置くと、`/ファイル名` で呼び出せるエージェントになる。

例えば、こういうファイルを作る。

```markdown
---
name: pr-sns-manager
description: PR・SNS部門エージェント。プレスリリース・SNS運用を担当する。
model: claude-sonnet-4-6
---

あなたはPR・SNS部門のマネージャーです。

## 役割
- PR TIMESへのプレスリリース作成
- X（Twitter）・Instagram・LinkedInの投稿文作成
- メディアへのピッチ文作成
...
```

これを置いておくだけで、`/pr-sns-manager` と入力するとPR専門のエージェントとして動く。

自分は6つのスキルファイルを作った。PM・DX支援・EC広告・クリエイティブ・PR/SNS・開発。それぞれが専門家として振る舞う。

---

## 実際の使い方

こんな感じで使う。

**例1：SNS投稿を作りたい**
```
/pr-sns-manager
新商品のプレスリリースを書いてください。
商品名：○○、価格：○○円、発売日：○月○日
```

→ PR専門エージェントが起動し、PR TIMES向けのプレスリリース文を作成してくれる。

**例2：よくわからない案件をとりあえずPMに投げる**
```
/pm
クライアントから「売上を上げたい」と言われています。
どの部門が対応すべきか整理してください。
```

→ PMエージェントがヒアリングを行い、担当部門を特定して案件ブリーフを作成する。

**例3：自然言語でも起動する**
```
プレスリリースを書いてほしい
```

→ CLAUDE.mdに「このキーワードが来たらPR部門エージェントを起動する」と書いておくだけで、スラッシュコマンドなしでも対応できる。

---

## CLAUDE.mdで「会社のルール」を定義する

面白いのが、CLAUDE.mdには組織のルールも書けることだ。

```markdown
## 重要ルール
- すべての応答・確認・質問は日本語で行う
- 作業前に不明点があれば必ず確認してから進む
- 成果物は projects/[部門名]/ に格納する

## 自動具体化ルール（常時適用）
ユーザーの入力が抽象的な場合、自動で具体化の質問を行う。
- 5W1Hのうち2つ以上が不明なら質問する
- 1回の返答につき質問は1〜2個に絞る
```

これを書いておくと、「なんとなくLP作って」という曖昧な指示が来たとき、AIが自動で「ターゲットは誰ですか？」「目的は認知獲得ですか、コンバージョンですか？」と聞いてくる。

曖昧な指示を受け取って曖昧な成果物を出す問題を、ルールで解決できる。

---

## なぜこれが価値あるのか

**1. コンテキストを毎回説明しなくていい**

普通にClaude（web版）を使うと、毎回「あなたはPR担当です。今回の案件は…」と説明しなければならない。スキルファイルに定義しておけば、その説明が不要になる。

**2. 複数の専門家を切り替えられる**

「LP文言はクリエイティブ部門」「LP実装は開発部門」のように、専門ごとに使い分けられる。同じ案件でも異なる視点で処理される。

**3. チームの知識を蓄積できる**

スキルファイルに「この部門はこういう判断基準で動く」を書いておけば、実際のビジネス知識がファイルに蓄積される。新しいプロジェクトでも同じ知識を引き継げる。

---

## この仕組みを作るのにかかった時間

CLAUDE.md：約2時間
スキルファイル6本：各30分×6 = 3時間

**合計5時間で、6人の専門家チームが手に入った。**

---

## まとめ

Claude Codeは「コードを書くツール」ではなく「AIチームを設計するツール」だった。

- **CLAUDE.md**：会社の定義・ルールを書く
- **スキルファイル**：専門家エージェントを定義する
- **自然言語トリガー**：コマンドなしで専門家を呼び出す

この3つの組み合わせで、自分一人でも多部門のビジネス支援ができる仕組みができた。
