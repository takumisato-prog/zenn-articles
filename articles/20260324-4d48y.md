## フック機能を知らないまま使うのはもったいない

Claude Codeを使い込んでいると、「この操作の前に毎回テストを走らせたい」「ファイル保存後に自動でlintをかけたい」という欲求が必ず出てくる。

私はClaude Codeで47名のAIエージェントが稼働する会社「Altus」を運営しているが、フック機能を導入する前と後では、自動化できる範囲がまるで違った。体感で5倍は広がっている。スキル機能やCLAUDE.mdが「AIに何をさせるか」を定義するものだとすれば、フック機能は「AIの動作の前後に何を挟むか」を制御する仕組みである。

この記事では、フック機能の基本概念から、Altusで実際に運用している設定まで、すべてコード付きで解説する。

## フック（hooks）とは何か

フックとは、Claude Codeの特定のイベント（アクション）が発生したタイミングで、任意のシェルコマンドを自動実行する仕組みである。

設定は `.claude/settings.json` もしくはプロジェクトの `settings.json` に記述する。

```json
// .claude/settings.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write",
        "hooks": [
          {
            "type": "command",
            "command": "echo 'ファイル書き込みが実行されます'"
          }
        ]
      }
    ]
  }
}
```

これだけで、Claudeがファイルを書き込む直前にコマンドが走る。単純だが、この「任意のシェルコマンド」という自由度が極めて強力だ。

## 使えるイベントは4種類

フックで指定できるイベントは以下の4つである。

| イベント名 | 発火タイミング | 主な用途 |
|---|---|---|
| `PreToolUse` | ツール実行の直前 | バリデーション、確認、前処理 |
| `PostToolUse` | ツール実行の直後 | ログ記録、後処理、通知 |
| `Notification` | Claude Codeが通知を送る時 | 外部通知連携 |
| `Stop` | Claudeがレスポンス生成を完了した時 | 最終チェック、レポート生成 |

`PreToolUse` と `PostToolUse` にはさらに `matcher` を指定でき、特定のツール（`Write`, `Edit`, `Bash`, `Read` など）に絞って発火させられる。

## 実践：Altusで稼働している5つのフック設定

### 1. ファイル書き込み前の自動バリデーション

AIエージェントがコードを書く前に、対象ディレクトリの権限を確認する。Altusでは47エージェントが並行稼働するため、誤って他チームのファイルを上書きする事故を防ぐ目的で導入した。

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write",
        "hooks": [
          {
            "type": "command",
            "command": "python3 scripts/validate_write_permission.py \"$CLAUDE_FILE_PATH\""
          }
        ]
      }
    ]
  }
}
```

```python
# scripts/validate_write_permission.py
import sys
import json
import os

file_path = sys.argv[1] if len(sys.argv) > 1 else ""
ALLOWED_DIRS = ["src/", "tests/", "docs/"]

allowed = any(file_path.startswith(d) for d in ALLOWED_DIRS)

if not allowed:
    result = {"decision": "block", "reason": f"書き込み禁止ディレクトリ: {file_path}"}
else:
    result = {"decision": "approve"}

print(json.dumps(result))
```

**ポイント：** `PreToolUse` フックが `{"decision": "block"}` を返すと、そのツール実行がキャンセルされる。これがフック最大の特徴だ。承認制御をプログラマブルに実装できる。

### 2. Bashコマンド実行後の自動ログ記録

エージェントが実行したすべてのBashコマンドをログファイルに記録する。障害発生時の原因追跡に不可欠だ。

```json
{
  "PostToolUse": [
    {
      "matcher": "Bash",
      "hooks": [
        {
          "type": "command",
          "command": "echo \"$(date -u '+%Y-%m-%dT%H:%M:%SZ') | BASH | $CLAUDE_SESSION_ID\" >> logs/agent_commands.log"
        }
      ]
    }
  ]
}
```

### 3. 完了時にSlack通知を飛ばす

エージェントがタスクを完了したら、即座にSlackに通知する。Altusではこれが運用の生命線になっている。

```json
{
  "Stop": [
    {
      "hooks": [
        {
          "type": "command",
          "command": "scripts/notify_slack.sh"
        }
      ]
    }
  ]
}
```

```bash
#!/bin/bash
# scripts/notify_slack.sh
WEBHOOK_URL="https://hooks.slack.com/services/XXXXX"
AGENT_NAME="${CLAUDE_AGENT_NAME:-unknown}"
TIMESTAMP=$(date '+%Y-%m-%d %H:%M:%S')

curl -s -X POST "$WEBHOOK_URL" \
  -H 'Content-Type: application/json' \
  -d "{\"text\": \"✅ エージェント *${AGENT_NAME}* がタスクを完了しました (${TIMESTAMP})\"}"
```

### 4. テスト自動実行フック

ファイル書き込み後に、変更されたファイルに対応するテストを自動で走らせる。

```json
{
  "PostToolUse": [
    {
      "matcher": "Write",
      "hooks": [
        {
          "type": "command",
          "command": "scripts/run_related_tests.sh"
        }
      ]
    }
  ]
}
```

```bash
#!/bin/bash
# scripts/run_related_tests.sh
FILE_PATH="${CLAUDE_FILE_PATH}"

if [[ "$FILE_PATH" == src/*.ts ]]; then
  TEST_FILE="${FILE_PATH/src\//tests/}"
  TEST_FILE="${TEST_FILE/.ts/.test.ts}"
  if [[ -f "$TEST_FILE" ]]; then
    npx vitest run "$TEST_FILE" --reporter=json 2>/dev/null | tail -1
  fi
fi
```

### 5. Notification転送でモバイル監視

Claude Codeの通知イベントをntfyに転送し、スマホで全エージェントの状態を監視する。

```json
{
  "Notification": [
    {
      "hooks": [
        {
          "type": "command",
          "command": "curl -s -d \"Claude Code: ${CLAUDE_NOTIFICATION_MESSAGE}\" ntfy.sh/altus-agents"
        }
      ]
    }
  ]
}
```

これにより外出中でもエージェントの異常停止やエラーを即座に検知できる。

## フック設計で押さえるべき3原則

Altusで47エージェントを運用する中で見えてきた原則がある。

**1. フックは軽くしろ**

フックのコマンドが重いと、Claude Code全体のレスポンスが遅延する。特に `PreToolUse` は同期実行されるため、処理は1秒以内に収めるべきだ。重い処理はバックグラウンドで実行する。

```bash
# 悪い例：同期でテストスイート全体を実行
npm test

# 良い例：バックグラウンドで実行し、結果はログに
npm test > logs/test_result.log 2>&1 &
```

**2. matcher は狭く絞れ**

すべてのツール実行にフックをかけると、`Read`（ファイル読み込み）のような頻繁に発火するイベントでも毎回コマンドが走る。matcherで対象を絞ることがパフォーマンスと安定性の鍵だ。

**3. エラーハンドリングを入れろ**

フックスクリプトがエラーで異常終了すると、Claude Code側の挙動に影響が出る場合がある。必ず `set -e` を避け、適切にエラーを捕捉する。

```bash
#!/bin/bash
# フックスクリプトのテンプレート
result=$(some_command 2>/dev/null) || true
if [[ -n "$result" ]]; then
  echo "$result"
fi
```

## CLAUDE.mdやスキルとの違い

混同しやすいので整理する。

| 機能 | 役割 | 制御対象 |
|---|---|---|
| CLAUDE.md | AIの振る舞い・ルール定義 | プロンプトレベル |
| スキル | 専門知識・手順の定義 | AIの知識レベル |
| **フック** | **イベント駆動の自動処理** | **システムレベル** |

CLAUDE.mdでどれだけ「テストを実行してからコミットしろ」と書いても、AIが忘れることはある。フックなら仕組みとして強制できる。これが決定的な違いだ。

## まとめ：フックは「AIの行動を仕組みで管理する」唯一の方法

フック機能の本質は、AIエージェントの行動を「お願い」ではなく「仕組み」で制御できる点にある。

Altusの運用でも、フック導入前は「エージェントがテストを飛ばした」「本番ブランチに直接pushした」といったヒヤリハットが月に数回あった。フック導入後、これらはゼロになった。

Claude Codeを単なる対話ツールとして使うか、自動化基盤として使うか。その分岐点がフック機能である。
