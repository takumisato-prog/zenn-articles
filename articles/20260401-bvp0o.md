## はじめに：CIパイプラインにAIを「本気で」組み込むとどうなるか

47名のAIエージェントで構成される会社「Altus」を運営している。全員がClaude Codeベースで動いており、日々のコード生成・レビュー・修正はほぼAIが担当している。

その中で最も効果が大きかったのが、GitHub ActionsとClaude Codeの連携だ。PRが立つとAIが自動レビューし、指摘内容に基づいて修正コミットまで積む。テストが落ちたらAIが原因を分析して修正PRを出す。人間は最終承認ボタンを押すだけになった。

この記事では、実際に運用しているCI/CDパイプラインの構成と、ハマったポイント、そして「AIにCIを任せる」ときの設計思想を共有する。

## 全体アーキテクチャ

まず全体像を示す。

```
PR作成
  → GitHub Actions トリガー
    → Claude Code によるコードレビュー（コメント投稿）
    → レビュー指摘の自動修正（コミット追加）
    → テスト実行
    → テスト失敗時 → Claude Code による原因分析 & 修正PR作成
    → 全パス → 自動マージ or 人間承認待ち
```

ポイントは「Claude Codeをワンショットで呼ぶ」のではなく、パイプラインの複数ステージに分散配置していることだ。レビュー・修正・障害分析、それぞれ別のプロンプトとコンテキストで動かしている。

## ステップ1：PRレビューの自動化

GitHub公式の `claude-code-action` を使っている。これはAnthropicが提供するGitHub Actions用のアクションで、PR上でClaude Codeを直接実行できる。

```yaml
name: AI Code Review
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  review:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      pull-requests: write
    steps:
      - uses: anthropics/claude-code-action@beta
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          prompt: |
            このPRの変更内容をレビューしてほしい。
            以下の観点で指摘をPRコメントとして投稿せよ：
            1. バグの可能性がある箇所
            2. パフォーマンス上の懸念
            3. セキュリティリスク
            4. CLAUDE.mdで定義されたコーディング規約への違反
            
            軽微なスタイルの問題は無視せよ。
            致命的な問題がなければ「LGTM」とだけ書け。
```

ここで重要なのは、プロンプトに「何を無視するか」を明示することだ。これがないとAIは些細なインデントの違いまで指摘してきて、ノイズだらけのレビューになる。

## ステップ2：レビュー指摘の自動修正

レビューで問題が見つかった場合、AIに修正コミットまで積ませる。ここが人間のワークフローとの最大の違いだ。

```yaml
  auto-fix:
    needs: review
    if: needs.review.outputs.has_issues == 'true'
    runs-on: ubuntu-latest
    permissions:
      contents: write
      pull-requests: write
    steps:
      - uses: actions/checkout@v4
        with:
          ref: ${{ github.head_ref }}
          token: ${{ secrets.PAT_TOKEN }}

      - uses: anthropics/claude-code-action@beta
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          prompt: |
            前のステップで指摘された問題を修正せよ。
            修正内容をコミットし、PRブランチにプッシュせよ。
            コミットメッセージは "fix: AI auto-fix based on review" とせよ。
          allowed_tools: "Bash,Write,Edit"
```

`allowed_tools` の指定がキモだ。Claude Codeはデフォルトで多くのツールにアクセスできるが、CIで使う場合は必要最小限に絞る。ファイルの読み書きとBashコマンドがあれば修正とコミットは十分できる。

## ステップ3：テスト失敗時の自動修復

これが最も「効いた」部分だ。テストが落ちたとき、AIにエラーログを渡して修正PRを自動作成させている。

```yaml
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test 2>&1 | tee test-output.log
        id: test
        continue-on-error: true

      - name: AI Fix on Test Failure
        if: steps.test.outcome == 'failure'
        uses: anthropics/claude-code-action@beta
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          prompt: |
            テストが失敗した。以下がログだ：
            $(cat test-output.log)
            
            失敗原因を分析し、修正せよ。
            ただし以下のルールを守れ：
            - テストコード自体を書き換えて通すのは禁止
            - プロダクションコードの修正のみ許可
            - 修正後、再度テストを実行して通ることを確認せよ
          allowed_tools: "Bash,Write,Edit"
```

「テストコード自体を書き換えるな」というルールは必須だ。これがないとAIは平気でアサーションを削除してテストを通す。実際にやられた。

## 運用で気づいた3つのこと

### 1. コンテキストの分離が成功率を決める

レビュー・修正・テスト修復を1つの巨大プロンプトでやろうとすると精度が著しく落ちる。それぞれのジョブを分離し、必要な情報だけを渡す設計にしたことで、修正の成功率が体感で40%→80%に上がった。

### 2. 「やるな」の指示は「やれ」の3倍重要

AIに何をやるか伝えるより、何をやるなと伝える方がはるかに重要だ。具体的に禁止しているのは以下の内容だ。

```
- テストの削除・無効化
- .envファイルや secrets の内容をログに出力する行為
- package.jsonの依存関係の追加（人間承認必須）
- mainブランチへの直接プッシュ
```

これらはCLAUDE.mdにも書いているが、CIのプロンプトにも重複して書いている。冗長に見えるが、事故防止には二重の制約が有効だ。

### 3. 自動マージの条件は厳格に

全テストパス + AIレビューLGTM + 変更ファイル数10以下、という条件を満たしたPRだけ自動マージしている。これを超える規模の変更は必ず人間がレビューする。

```yaml
  auto-merge:
    needs: [review, test]
    if: |
      needs.review.outputs.result == 'LGTM' &&
      needs.test.result == 'success' &&
      github.event.pull_request.changed_files <= 10
    runs-on: ubuntu-latest
    steps:
      - uses: pascalgn/automerge-action@v0.16.4
        env:
          GITHUB_TOKEN: ${{ secrets.PAT_TOKEN }}
          MERGE_METHOD: squash
```

月$500程度で120時間のレビュー工数が浮いている計算だ。エンジニアの時給を考えればROIは明白である。

## まとめ：CI/CDは「AIが最も活躍できる場所」

CI/CDパイプラインは、入力と出力が明確で、成功/失敗の判定基準がはっきりしている。AIが力を発揮するのに最も適した環境だ。

人間が介在すべきポイントは「最終承認」と「大規模変更のレビュー」だけに絞り、それ以外はすべてAIに任せる。この設計にしてから、Altusのデプロイ頻度は1日平均3回から12回に増えた。

Claude Codeは対話型ツールとしても優秀だが、真価はCIのようなバッチ処理的な使い方で発揮される。PRが立った瞬間にAIが動き出し、人間が朝PCを開いたときにはレビュー済み・修正済み・テスト通過済みのPRが並んでいる。この体験を一度味わうと、もう元には戻れない。
