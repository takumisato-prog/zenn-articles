## なぜ47名なのか

AIエージェント会社「Altus」を運営して数ヶ月が経った。よく聞かれるのが「なぜ47名なのか」という質問である。

答えはシンプルだ。**実際の業務を回すために必要な最小人数が47名だった**。最初から47名を想定したわけではない。業務を分解し、役割を定義し、1つずつエージェントを追加していった結果がこの数字である。

10名では足りなかった。30名でもまだ隙間があった。47名で初めて「誰に振ればいいかわからない仕事」がなくなった。

本記事では、既存記事で触れたスキルファイルの書き方やCLAUDE.mdの設定ではなく、**47名をどういう思想で配置し、どう連携させているか**という組織設計の全容を解説する。

## 5部門・12チームの組織構造

Altusの組織は以下の5部門で構成されている。

```
Altus 組織図（47名体制）
├── 経営企画部（5名）
│   ├── 戦略チーム（2名）: CEO / 戦略アナリスト
│   └── 管理チーム（3名）: CFO / 法務 / 総務
├── プロダクト開発部（15名）
│   ├── フロントエンドチーム（4名）: リード / React専門 / UI/UX / アクセシビリティ
│   ├── バックエンドチーム（4名）: リード / API設計 / DB設計 / インフラ
│   ├── QAチーム（3名）: テスト設計 / 自動テスト / セキュリティ
│   └── アーキテクトチーム（4名）: システム設計 / DX推進 / パフォーマンス / 技術調査
├── コンテンツ部（12名）
│   ├── ライティングチーム（4名）: 編集長 / テックライター / コピーライター / 校正
│   ├── デザインチーム（4名）: アートディレクター / グラフィック / 動画 / ブランド
│   └── SEO・分析チーム（4名）: SEO / データ分析 / 市場調査 / 競合分析
├── 営業・CS部（9名）
│   ├── セールスチーム（4名）: 営業部長 / 提案書作成 / 見積 / CRM管理
│   └── CSチーム（5名）: CS責任者 / オンボーディング / サポート / FAQ / VoC分析
└── 人事・組織開発部（6名）
    ├── 採用チーム（3名）: 採用責任者 / 評価 / 育成
    └── 組織チーム（3名）: カルチャー / ナレッジ管理 / プロセス改善
```

「AIに人事部は要らないだろう」と思うかもしれない。だが、**新しいエージェントを追加するときの役割定義、既存エージェントのスキルファイル更新、パフォーマンス評価に基づく改善**——これらはまさに「人事」の仕事である。

## 設計の3原則

47名の配置を決めるにあたり、3つの原則を設けた。

### 原則1: 1エージェント＝1責務

最も重要な原則がこれだ。人間の組織でも「何でも屋」は属人化の温床になる。AIエージェントも同じである。

```markdown
# 悪い例：責務が曖昧
あなたはフロントエンドとバックエンドの両方を担当するフルスタックエンジニアです。
テストも書いてください。デプロイもお願いします。

# 良い例：責務が明確
あなたはReact専門のフロントエンドエンジニアです。
担当範囲：UIコンポーネントの実装、状態管理設計、コンポーネントテスト
担当外：API実装、インフラ構築、E2Eテスト（それぞれ専任がいる）
```

「担当外」を明記するのがコツである。AIは親切なので、頼まれたら何でもやろうとする。境界を明示しないと責務が崩壊する。

### 原則2: 3層のエスカレーション構造

すべてのエージェントは「メンバー → チームリード → 部門長」の3層構造に属する。問題が発生したとき、誰に相談するかが決まっている。

```markdown
# チームリード用スキルファイルの一部
## エスカレーションルール
- メンバーから判断を求められた場合：チーム内で解決を試みる
- 他チームとの調整が必要な場合：部門長に報告し指示を仰ぐ
- 事業方針に関わる判断の場合：CEOに直接エスカレーション

## 配下メンバー
- React専門（@frontend-react）
- UI/UX（@frontend-uiux）
- アクセシビリティ（@frontend-a11y）
```

実際の運用では、Claude Codeのセッション内でこのエスカレーション構造を「会話のフロー」として再現する。1つのセッションで複数エージェントの視点を切り替えながら作業を進める形だ。

### 原則3: インターフェース駆動の連携

エージェント間の連携は、自然言語の「お願い」ではなく、**構造化されたフォーマット**で行う。

```markdown
## チーム間連携フォーマット

### 開発→QAへのテスト依頼
```json
{
  "type": "test_request",
  "from": "backend-lead",
  "to": "qa-lead",
  "target": "src/api/users.ts",
  "changes": ["新規エンドポイント追加", "バリデーション変更"],
  "priority": "high",
  "deadline": "当日中"
}
```

### QA→開発へのバグ報告
```json
{
  "type": "bug_report",
  "from": "qa-testing",
  "to": "backend-lead",
  "severity": "critical",
  "reproduction": "POST /api/users に空文字列を送信",
  "expected": "400エラー",
  "actual": "500エラー"
}
```
```

このフォーマットを統一したことで、**依頼の抜け漏れが激減した**。人間同士の曖昧なSlackメッセージと違い、AIは構造化データを正確に処理できる。

## 実際の業務フロー：記事制作の場合

1本の技術記事を作る場合、以下のエージェントが順に稼働する。

| 順序 | エージェント | 作業内容 |
|------|-------------|---------|
| 1 | 市場調査 | トレンド分析、検索ボリューム調査 |
| 2 | SEO | キーワード選定、タイトル候補作成 |
| 3 | 編集長 | 企画承認、構成案レビュー |
| 4 | テックライター | 本文執筆 |
| 5 | 校正 | 誤字脱字、事実確認、トーン統一 |
| 6 | コピーライター | タイトル・見出しのブラッシュアップ |
| 7 | データ分析 | 公開後のPV・読了率追跡 |

7名が関わるが、**1つのClaude Codeセッションの中で順にペルソナを切り替えて実行する**。具体的には、以下のようにプロンプトで指示する。

```
今から記事制作フローを開始する。
まず「市場調査担当」として、"Claude Code エージェント設計"に関する
最新トレンドと検索需要を分析してくれ。
完了したら「SEO担当」に切り替えて、キーワード選定に移る。
```

このフローをスキルファイルにテンプレート化してあるため、毎回指示を書く必要はない。`/skill:content-pipeline` と呼び出すだけで全工程が走る。

## 47名体制で見えてきた限界と対策

### 限界1: コンテキストウィンドウの圧迫

47名分の役割定義をすべて1セッションに読み込むのは現実的ではない。対策として、**その作業に関係するエージェントだけをロードする仕組み**にした。

```
altus/
├── skills/
│   ├── departments/
│   │   ├── engineering/    # 開発部のスキル群
│   │   ├── content/        # コンテンツ部のスキル群
│   │   └── ...
│   └── workflows/
│       ├── article-pipeline.md    # 記事制作フロー
│       ├── feature-development.md # 機能開発フロー
│       └── client-onboarding.md   # 顧客対応フロー
```

ワークフロー単位でスキルをバンドルし、必要なエージェント定義だけをコンテキストに含める。

### 限界2: エージェント間の矛盾

SEO担当が「このキーワードを入れろ」と言い、校正担当が「不自然だから削除しろ」と言う。こういった矛盾は頻繁に起きる。

対策は**優先度の明文化**だ。

```markdown
## 判断基準の優先順位（コンテンツ部）
1. 事実の正確性（校正が最終判断権を持つ）
2. 読者体験（編集長が最終判断権を持つ）
3. SEO最適化（SEO担当が提案、編集長が承認）
```

「誰が最終決定権を持つか」をスキルファイルに書いておくだけで、矛盾の大半は解消される。

## 組織設計から得た最大の学び

47名のAI組織を設計して最も強く実感したのは、**「AIの性能」よりも「役割設計の精度」が成果を決める**ということだ。

同じClaude 4 Sonnetでも、「何でもやって」と言ったときと「あなたはAPI設計の専門家で、RESTful設計原則に基づいてエンドポイントを設計する。他の領域には口出ししない」と言ったときでは、出力の質がまるで違う。

これは人間の組織と完全に同じ構造である。優秀な人材を集めても、役割が曖昧なら組織は機能しない。AIエージェントも同様に、**組織設計こそが最大のレバレッジポイント**である。

47名が多いか少ないかは、事業の規模と複雑さによる。重要なのは数ではなく、**全員の責務に隙間と重複がないこと**だ。この原則さえ守れば、5名でも100名でも機能する組織を作れる。
