## はじめに──AIは「忘れる」という致命的な問題

Claude Codeを使い込んでいると、必ずぶつかる壁がある。**会話が切れるとすべてを忘れる**という問題だ。

私はClaude Codeで47名のAIエージェント会社「Altus」を運営しているが、初期の頃はこれに本当に苦しんだ。朝イチでエージェントに「昨日の続きをやって」と指示すると、「昨日の作業について教えてください」と返ってくる。毎回コンテキストを説明し直す時間が、1エージェントあたり5〜10分。47体ともなると、それだけで1日が終わる。

この問題を解決したのが、Claude Codeの**メモリ機能**だ。CLAUDE.mdとは別物で、会話をまたいだ記憶の永続化を実現する仕組みである。本記事では、既存記事で触れたCLAUDE.mdやスキル機能とは異なる「メモリ機能そのもの」に焦点を当て、47体運用で培った実践ノウハウを共有する。

## メモリ機能とは何か──CLAUDE.mdとの決定的な違い

まず整理しておきたい。Claude Codeにおける「記憶」に関わる仕組みは複数ある。

| 仕組み | 永続性 | 用途 | 誰が書くか |
|--------|--------|------|------------|
| CLAUDE.md | 永続 | プロジェクトのルール・方針 | 人間が事前に書く |
| スキルファイル | 永続 | 特定タスクの手順書 | 人間が事前に書く |
| **メモリ機能** | 永続 | 会話中に得た知識・判断の記録 | **AIが自分で書く** |

CLAUDE.mdが「会社の就業規則」、スキルファイルが「業務マニュアル」だとすれば、メモリ機能は**「個人のノート」**に相当する。AIが会話の中で学んだこと、判断したこと、作業の進捗を自ら記録し、次回の会話で自動的に読み込む。

具体的には、会話中にClaude Codeへ以下のように指示する。

```
この判断を覚えておいて
```

すると、Claude Codeは `~/.claude/` 配下のメモリファイルに情報を書き込む。次の会話開始時にこのファイルが自動的に読み込まれ、過去の文脈を踏まえた応答が可能になる。

## 実践：メモリ機能を「作業継続」に使う3つのパターン

### パターン1：作業進捗の自動記録

47体のエージェントを運用する上で最も効果が大きかったのが、作業終了時に進捗を自動記録させるパターンだ。

会話の最後に以下のように伝える。

```
今日の作業内容と、明日やるべきことをメモリに記録して。
未解決の問題があればそれも含めて。
```

するとClaude Codeは、こんな形でメモリに記録を残す。

```markdown
# 作業記録 2025-01-15

## 完了タスク
- APIエンドポイント /users/profile のバリデーション修正
- テストケース12件追加（全件パス）

## 未完了・次回対応
- /users/settings のレスポンス形式がフロントと不整合（Issue #234）
- パフォーマンステスト未実施

## 判断メモ
- date-fnsではなくdayjsを採用した（バンドルサイズ削減のため）
- エラーコード体系はHTTPステータスに準拠する方針とした
```

翌日の会話で「昨日の続きをやって」と言えば、AIはこの記録を読み込んだ上で、Issue #234の調査から始めてくれる。

### パターン2：判断履歴の蓄積

プロジェクトが進むと、「なぜこの技術を選んだのか」「なぜこの設計にしたのか」という判断の文脈が重要になる。メモリ機能で判断履歴を蓄積させると、一貫性のある開発が可能になる。

```
今の設計判断をメモリに記録して。
理由と、却下した代替案も含めて残しておいて。
```

記録例：

```markdown
# 設計判断 2025-01-15

## 認証方式の選定
- **採用**: JWT + リフレッシュトークン
- **却下**: セッションベース認証（水平スケール時の複雑性）
- **却下**: OAuth2のみ（自社サービス間通信には過剰）
- **理由**: マイクロサービス間の認証伝播が容易。
  リフレッシュトークンのローテーション実装で安全性も確保。
```

これがあると、2週間後に別のエージェントが関連タスクを担当しても、「なぜJWTなのか」を理解した上で実装を進められる。

### パターン3：エージェント間の引き継ぎメモ

Altusでは、1つの機能開発を複数のエージェントがリレー形式で担当することがある。フロントエンド担当がUI実装を終えた後、バックエンド担当が連携部分を実装する、といった具合だ。

このとき、作業を終えたエージェントに以下を指示する。

```
次にこのコードベースを触るエージェントへの引き継ぎメモをメモリに残して。
ハマりポイントと、絶対に変えてはいけない部分を明記して。
```

記録例：

```markdown
# 引き継ぎメモ: ダッシュボード機能

## ハマりポイント
- ChartコンポーネントはuseEffectの依存配列を空にすると
  リサイズイベントが拾えなくなる。必ず[windowSize]を入れること。
- APIレスポンスのdataフィールドがnullの場合がある（データ未生成時）。
  フロント側でフォールバック表示を入れ済み。

## 変更禁止
- /components/Chart/index.tsx のrefの取り回し。
  Canvas APIとReactのライフサイクルを同期させる処理で、
  安易に変えるとメモリリークが起きる。
```

## 運用で気づいた3つの落とし穴

### 落とし穴1：メモリが肥大化して逆効果になる

何でも記録させると、メモリファイルが膨大になり、読み込み時にコンテキストウィンドウを圧迫する。Altusでは初期にこれをやらかして、メモリ読み込みだけでトークンの20%を消費するエージェントが出た。

**対策**：週に1回、メモリの棚卸しを行う。

```
メモリの内容を確認して、完了済みかつ今後参照しないものを削除して。
設計判断の記録は残し、作業進捗の記録は完了分を消して。
```

### 落とし穴2：メモリの内容が矛盾する

複数のセッションで作業すると、古いメモリの記述と最新のコードが矛盾することがある。AIは律儀にメモリの記述を信じるので、「メモリには未実装と書いてあるが、実際には実装済み」という状態で混乱が起きた。

**対策**：作業開始時に必ず「メモリの内容と現在のコードを突合して、矛盾があれば報告して」と指示する。これをCLAUDE.mdの起動時ルールに書いておくと自動化できる。

### 落とし穴3：メモリに機密情報を書かせてしまう

APIキーやパスワードがメモリに記録されるケースがあった。メモリファイルはプレーンテキストなので、これは危険だ。

**対策**：CLAUDE.mdに以下を明記する。

```markdown
## メモリ記録のルール
- APIキー、パスワード、トークンなどの秘密情報は絶対にメモリに記録しない
- 環境変数名の記録は許可する（例：DATABASE_URLを使用）
- 個人情報に該当するデータは記録しない
```

## 47体運用で確立したメモリ管理のベストプラクティス

最後に、現在Altusで運用しているメモリ管理のルールをまとめる。

```markdown
# メモリ運用ポリシー

## 記録すべきもの
- 設計判断とその理由
- 未解決のバグ・課題
- 他エージェントへの引き継ぎ事項
- プロジェクト固有の命名規則や制約

## 記録しないもの
- コードそのもの（ファイルパスの参照で十分）
- 一般的な技術知識（AIが元から知っている）
- 機密情報

## メンテナンスサイクル
- 毎セッション終了時：進捗を記録
- 毎週月曜：完了済みタスクの記録を削除
- 毎月初：設計判断の記録を見直し、陳腐化したものを更新
```

## まとめ

メモリ機能は、Claude Codeを「使い捨ての対話ツール」から「継続的に学習する作業パートナー」に変える仕組みだ。CLAUDE.mdが会社のルールブック、スキルファイルが業務マニュアルだとすれば、メモリは**経験の蓄積**そのものである。

47体のエージェントを運用していて確信しているのは、AIの生産性を決めるのはモデルの性能ではなく、**コンテキストの質**だということだ。メモリ機能を正しく運用すれば、毎回ゼロから説明し直す無駄がなくなり、AIの出力品質が劇的に向上する。

まずは1体のエージェントで「作業終了時に進捗を記録させる」ところから始めてみてほしい。翌日の「昨日の続きをやって」が通じた瞬間、AIとの協業の質が一段変わるはずだ。
