## 人間のコードレビューは、もう限界だった

AIエージェント会社「Altus」では、47名のAIエージェントが日々コードを書いている。当然、そのコードには誰かのレビューが必要だ。

最初は自分（人間）がすべてレビューしていた。だが、1日に上がってくるプルリクエストが30件を超えたあたりで破綻した。レビュー待ちの行列ができ、マージまでの平均リードタイムは18時間。見落としも増え、本番環境でのバグ発生率が週あたり4.2件にまで膨らんだ。

そこで「AIエージェントがAIエージェントのコードをレビューする」体制を構築した。結果、バグ発生率は週1.8件に半減し、レビューのリードタイムは平均42分にまで短縮された。この記事では、その具体的な仕組みと、運用してわかった勘所を共有する。

## レビュー専任エージェント「Auditor」の設計思想

Altusには役割ごとにエージェントがいるが、コードレビュー専任として3名の「Auditor」エージェントを配置している。重要なのは、**コードを書くエージェントとレビューするエージェントを完全に分離した**ことだ。

なぜ分離するのか。人間の組織でも同じだが、自分が書いたコードの問題点には自分では気づきにくい。AIも同様で、同一コンテキストの中で生成と検証を同時にやらせると、自己正当化バイアスに近い挙動が頻発する。別エージェントに渡すことで、新鮮な視点からの検証が機能する。

Auditorに渡しているレビュー指示の骨格はこうなっている：

```markdown
# Auditor Review Protocol

## レビュー観点（優先順）
1. **セキュリティ**: インジェクション、認証漏れ、秘密情報のハードコード
2. **ロジックの正当性**: エッジケース、off-by-one、null安全性
3. **既存コードとの整合性**: 命名規則、アーキテクチャパターンの逸脱
4. **パフォーマンス**: N+1クエリ、不要なループ、メモリリーク候補
5. **テストの充足性**: 追加コードに対するテストカバレッジ

## 出力フォーマット
- 各指摘に severity を付与: critical / warning / suggestion
- critical が1件でもあれば REJECT
- warning のみなら条件付き APPROVE（修正後の再レビュー不要）
- suggestion のみなら APPROVE

## 禁則事項
- コードスタイルの好みだけの指摘は行わない
- 修正案を示さない指摘は行わない
```

## 実際のレビューフローを見せる

具体例で説明する。あるエージェント「Backend-03」がAPI認証ミドルウェアを修正したケースだ。

Backend-03が出力したコード：

```typescript
async function authenticateRequest(req: Request): Promise<User> {
  const token = req.headers.authorization?.split(' ')[1];
  const decoded = jwt.verify(token, process.env.JWT_SECRET);
  const user = await db.users.findById(decoded.sub);
  return user;
}
```

Auditorのレビュー結果：

```
## Review Result: REJECT

### [critical] token が undefined の場合に jwt.verify が例外を投げる
- 行: 3
- Authorization ヘッダーが未設定、または Bearer 形式でない場合、
  token は undefined になる。jwt.verify に undefined を渡すと
  JsonWebTokenError が発生し、500レスポンスになる。
- 修正案:
  ```typescript
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) {
    throw new AuthenticationError('Bearer token is required');
  }
  ```

### [critical] JWT_SECRET が undefined の場合の考慮がない
- 行: 3
- 環境変数が未設定の場合、空文字列で署名検証が行われ、
  セキュリティリスクとなる。
- 修正案: アプリケーション起動時にバリデーションするか、
  この関数内で明示的にチェックする。

### [warning] user が null の場合のハンドリングがない
- 行: 4
- トークンは有効だがユーザーが削除済みのケース。
- 修正案:
  ```typescript
  if (!user) {
    throw new AuthenticationError('User not found');
  }
  ```
```

critical が2件あるため REJECT。Backend-03は修正後に再提出し、Auditorが再レビューして APPROVE となった。このサイクルが平均42分で完了する。

## 品質が「2倍」と言える根拠

定量的に追跡している指標が3つある。導入前4週間と導入後4週間の比較データだ。

| 指標 | 導入前（週平均） | 導入後（週平均） | 変化率 |
|------|-----------------|-----------------|--------|
| 本番バグ発生数 | 4.2件 | 1.8件 | -57% |
| レビュー指摘で防げたcritical | 計測なし | 11.3件/週 | — |
| PRマージまでのリードタイム | 18時間 | 42分 | -96% |

「品質2倍」と表現したのは、本番バグ発生率がほぼ半分になった（＝同じコード量に対して2倍の品質を担保できている）ことに基づく。加えて、週あたり11件以上のcritical級の問題が本番到達前に止められている事実が大きい。

## 運用して気づいた3つの落とし穴

### 1. レビューが甘くなる「馴れ合い」問題

同じCLAUDE.mdを共有するエージェント同士だと、レビュー観点が画一化し、特定パターンの見落としが固定化する。対策として、Auditor 3名それぞれに異なる「重点観点」を持たせている。Auditor-Aはセキュリティ特化、Auditor-Bはパフォーマンス特化、Auditor-Cはアーキテクチャ整合性特化。PRごとにランダムに1名をアサインする。

### 2. 過剰な指摘による手戻りループ

suggestion レベルの指摘に対してもエージェントが律儀に全修正し、再レビューでまた新しい suggestion が出て……というループが発生した。解決策は前述のプロトコルにある通り、**suggestion のみなら即 APPROVE** というルールの厳格適用だ。

### 3. コンテキスト不足による誤検知

レビュー対象のdiff だけを渡すと、既存コードとの関係がわからず的外れな指摘が出る。これは`git diff` だけでなく、変更ファイルの全文と関連ファイル（import先など）を含めて渡すことで大幅に改善した。具体的には以下のようなコンテキスト収集スクリプトを使っている：

```bash
#!/bin/bash
# collect-review-context.sh
CHANGED_FILES=$(git diff --name-only main...HEAD)
for file in $CHANGED_FILES; do
  echo "===== $file ====="
  cat "$file"
  # import/requireしているローカルファイルも収集
  grep -oP "from ['\"]\..*?['\"]" "$file" | \
    sed "s/from ['\"]//;s/['\"]//" | \
    while read dep; do
      resolved=$(find . -path "*${dep}*" -name "*.ts" | head -1)
      if [ -n "$resolved" ]; then
        echo "----- dependency: $resolved -----"
        cat "$resolved"
      fi
    done
done
```

## 人間の役割はどこに残るのか

誤解のないように書いておくと、人間のレビューを完全に廃止したわけではない。以下の場合は自分がレビューする：

- **アーキテクチャレベルの設計変更**：ディレクトリ構成の変更、新しい外部サービスとの連携追加など
- **Auditor間で判断が割れたケース**：稀に、修正すべきかどうかの判断がAuditor間で矛盾することがある
- **週次のレビュー品質監査**：Auditorの指摘内容をサンプリングして、見落としパターンがないか確認する

人間がやるべきは「レビューそのもの」ではなく、「レビューの仕組みが正しく機能しているかの監査」だ。メタレビュー、とでも呼ぶべき役割にシフトしている。

## まとめ：レビューこそAIエージェントの最適用途

コード生成にAIを使う話は多い。だが、実運用で本当に効いたのはレビュー側だった。理由は明確で、レビューは「既存のコードに対して既知のパターンを照合する」タスクであり、AIの得意領域と完全に一致するからだ。

47名体制で回してわかったのは、**AIエージェントの品質は、AIエージェント同士のピアレビューで担保できる**ということだ。人間はその上位レイヤーで「仕組みの品質」を見ればいい。この構造が、スケーラブルなAI開発組織の鍵になると確信している。
